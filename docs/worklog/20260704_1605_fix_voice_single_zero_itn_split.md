## 任务概述

真机复报（截图）：说「五千三百一十二元」被识别为 **53102**；说「三千二百一十一元」**最后的 1 没有记录**（显示 3210）。转写本身即为「刚刚吃饭用了53102元」——毒化仍来自 iOS ITN，但这次是**单尾零**切分变体（`五千三百一十|二` → "5310"+"2"），落在 260703 三道防线的 ≥2 尾零保守阈值之外。

---

## 完成的工作

### 1. 根因

260703 的 `scan()` 位值合并（1C）本就支持单尾零（`_mergePositionalDigit` 对任意 scale>1 生效），但上游三道门都卡在 ≥2 零：

| 门 | 旧阈值 | 后果 |
|---|---|---|
| `detectConcatRepairCandidate` | head 尾零 ≥2 | 无空格 "53102" 不出修复候选 |
| `_spacedRoundGroupPattern` | 字面 `\d+00` | 带空格 "5310 2元" 不路由状态机（正则只剩尾段） |
| `_bufferLooksOpen` 整百规则 | `%100==0 && ≥100` | 分段 final "3210"+"1元"：缓冲判闭合 → 提交 3210、丢弃尾段（用户看到的「1 没记录」） |

### 2. 修复（三处阈值放宽到 ≥1 尾零）

- `voice_text_parser.dart`：检测器 `trailingZeros < 2` → `< 1`（tailLen ≤ 尾零数的配合约束保住精度：单零只收 1 位尾段）；路由正则 `\d+00` → `\d+0`（精确的"尾段是否装得下"由状态机合并强制，正则只做廉价形状门；日期负锚不变）。
- `voice_chunk_merger.dart`：开放判定 `%100==0 && ≥100` → `%10==0 && ≥10`（装不下的下一个数字仍走 last-wins 纠正语义）。

护栏保持：长度下限 5（「3005」三千零五不误报）、全零/零头尾段拒绝（「53100」「250000」不误报）、尾段 ≤2 位。

### 3. TDD

新增 13 条测试先 RED（`Expected [3211] / Actual [3210]`、`Expected 5312 / Actual 2`、`Expected 5312 / Actual null` 等）后 GREEN：machine（5310 2元→5312 已绿，证明 1C 无需改）、parser 路由 ×2、检测器 ×3、merger 分段 ×2、use case 候选+alternates 采用 ×2。
