## Key Files

### Entry Point & Bootstrapping
- `lib/main.dart` — single entry; performs master-key init, device-key init, encrypted DB creation, then runs `HomePocketApp`. There is **no** separate `AppInitializer` class despite CLAUDE.md mention.
- `lib/data/app_database.dart` — `@DriftDatabase` root, `AppDatabase.forTesting()` factory.
- `lib/infrastructure/crypto/database/encrypted_database.dart` — `ensureNativeLibrary()` (loads SQLCipher) and `createEncryptedExecutor(masterKeyRepo)`.

### Theme
- `lib/core/theme/app_theme.dart` — `MaterialApp.theme` definitions
- `lib/core/theme/app_colors.dart` — palette (`#8AB8DA`, `#47B88A`, `#5A9CC8`)
- `lib/core/theme/app_theme_colors.dart`
- `lib/core/theme/app_text_styles.dart` — `amountLarge/Medium/Small` with `FontFeature.tabularFigures()`

### State Management Hubs
Per feature, providers live under `lib/features/{feature}/presentation/providers/`:
- `repository_providers.dart` — concrete repository wiring (single source of truth per feature)
- `use_case_providers.dart` — use case wiring
- `*_notifier.dart` — class-style `@riverpod` notifiers
- `*.g.dart` — generated provider code (committed)

Cross-cutting providers (e.g., `appDatabaseProvider`, `currentLocaleProvider`) live alongside their domain — `appDatabaseProvider` in `lib/data/app_database.dart`, `currentLocaleProvider` in `lib/features/settings/presentation/providers/locale_provider.dart`.

### Database
- `lib/data/tables/` — 11 Drift tables: `audit_logs_table`, `books_table`, `categories_table`, `category_keyword_preferences_table`, `category_ledger_configs_table`, `group_members_table`, `groups_table`, `merchant_category_preferences_table`, `sync_queue_table`, `transactions_table`, `user_profiles_table`.
- `lib/data/daos/` — DAOs co-named with their tables (`book_dao`, `category_dao`, `transaction_dao`, ...) plus generated `.g.dart`.
- `lib/data/repositories/` — `*_repository_impl.dart`.

### i18n
- `lib/l10n/app_{en,ja,zh}.arb` — translation source files
- `lib/generated/app_localizations.dart` — `S` class (flutter_localizations codegen)
- `lib/infrastructure/i18n/formatters/date_formatter.dart` — locale-aware date formatting
- `lib/infrastructure/i18n/formatters/number_formatter.dart` — locale-aware currency formatting

## Naming Conventions (mapped to actual files)

| Concept | Convention | Examples |
|---|---|---|
| Dart source files | `snake_case.dart` | `transaction_form_screen.dart`, `key_manager.dart` |
| Test files | `*_test.dart`, mirror lib path | `delete_transaction_use_case_test.dart` |
| Generated files | `*.g.dart`, `*.freezed.dart`, `*.mocks.dart` | `repository_providers.g.dart`, `category.freezed.dart` |
| Drift table file | `*_table.dart` | `transactions_table.dart`, `audit_logs_table.dart` |
| Drift table class | `PascalCase` plural | `Transactions`, `AuditLogs` |
| Drift row class | `@DataClassName('PascalCaseRow')` | `TransactionRow`, `BookRow` |
| Drift index name | `idx_{table_short}_{columns}` | `idx_tx_book_id`, `idx_tx_book_timestamp` |
| DAO file / class | `*_dao.dart` / `*Dao` | `transaction_dao.dart` → `TransactionDao` |
| Repository interface | `*Repository` (in `domain/repositories/`) | `TransactionRepository` |
| Repository impl | `*RepositoryImpl` (in `data/repositories/`) | `TransactionRepositoryImpl` |
| Use case file / class | `*_use_case.dart` / `*UseCase` | `delete_transaction_use_case.dart` → `DeleteTransactionUseCase` |
| Riverpod notifier | `*Notifier extends _$Notifier` | `CategoryReorderNotifier` |
| Riverpod provider fn | `camelCase` `@riverpod` function | `bookRepository(Ref ref)` |
| Screens | `*_screen.dart` ending with `Screen` class | `TransactionFormScreen` |
| Widgets (feature) | `snake_case.dart`, `PascalCase` widget class | `amount_display.dart` → `AmountDisplay` |
| ARB keys | `camelCase` | `addTransaction`, `transactionList` |

### Directory Naming
- All directories are `lower_snake_case` (`family_sync`, `dual_ledger`, `infrastructure/crypto/services`).
- Feature directory names match domain in `application/{feature}/` (e.g., `application/accounting/` ↔ `features/accounting/`).
- Application-layer subdirs cover features that may not have a UI counterpart yet (`application/voice/` exists; there is no `features/voice/` since voice flows surface inside `accounting/`).

## Where to Put New Code (decision tree from CLAUDE.md)

1. Technology / platform capability → `lib/infrastructure/{topic}/`
2. Business logic / use case → `lib/application/{domain}/`
3. Drift table / DAO / repository impl → `lib/data/`
4. Domain model or repository interface → `lib/features/{feature}/domain/{models,repositories}/`
5. UI screen / widget / provider → `lib/features/{feature}/presentation/{screens,widgets,providers}/`
6. Cross-cutting helper used by 3+ features → `lib/shared/{constants,utils}/`
7. Theme / brand assets → `lib/core/theme/`
8. Default to `lib/{appropriate top-level dir}/` when uncertain

## Anti-patterns Already in Place to Avoid

- Don't add `application/`, `infrastructure/`, `data/tables/`, or `data/daos/` **inside** `lib/features/{feature}/` — the Thin Feature rule explicitly forbids it.
- Don't introduce barrel files (`index.dart`) — consumers import specific source files.
- Don't reach across feature boundaries from another feature's `presentation/`. Cross-feature collaboration goes through `application/` use cases, repositories, or shared infrastructure.
