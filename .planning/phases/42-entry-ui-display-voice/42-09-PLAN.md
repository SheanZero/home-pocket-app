<objective>
Build the detail/edit linked editing (DISP-03/DISP-04) under ADR-022 D-01 — the phase's #1 correctness watch item. The ROADMAP "three-field bidirectional" wording is VOID. The edit host shows THREE rows: original amount (editable) + rate (editable, never collapsed) + JPY (read-only derived via `convertToJpy()`). There is exactly ONE data-flow direction: original/rate → JPY. JPY is NEVER assigned as an input. "原币是事实，日元是结果." MUST NOT implement three-field bidirectional editing (circular-dependency risk — ADR-022 lines 37-55).

Wire the ADR-022 date-change semantics off the `RateSignal` the rate use case already pre-computes (do NOT recompute the >1% threshold in UI): D-02 (manual-override + date change → two-choice `ChangeRateConfirmationDialog`, NO default) and D-03 (no-override + >1% JPY change → non-blocking toast + Undo, 5s window). Surface both via `ref.listen`, NEVER `ref.watch` (RESEARCH Pitfall 4). Edit save persists the triple via the extended `UpdateTransactionUseCase` (42-03); hash NOT recomputed (ADR-021).

Also surface voice `detectedCurrency` (from 42-04) on this shared form, editable before save, triggering the normal rate-fetch flow (VOICE-CUR-02/03) — the form is reachable from the same `CurrencySelectorSheet` (VOICE-CUR-03).
