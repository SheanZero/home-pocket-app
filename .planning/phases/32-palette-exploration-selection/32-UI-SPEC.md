# Phase 32 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## ⚠️ Phase Nature — Read First

This is an **exploration-and-selection** UI phase that produces **ZERO production code**. Its only deliverables are:

1. **Mined-reference synthesis** (PALETTE-01) — written candidate color directions with rationale, drawn from VoltAgent/awesome-design-md brand `DESIGN.md` set + the dual-ledger family-finance context.
2. **4–5 full color-scheme proposals** (PALETTE-02) — each rendered as Pencil mockups across **3 representative screens** (home hero, transaction list, analytics) in **both light and dark**.
3. **Accepted ADR-018** (PALETTE-03) — records the single selected palette (or named hybrid) with exact hex per semantic role.

Because no single screen is being built, the conventional Typography / Spacing / Copywriting tables below are **frozen references to the existing Wa-Modern system**, NOT new declarations to redesign. Typography, spacing, corner-radius, and component geometry are **explicitly out of scope** for this phase (REQUIREMENTS.md Out-of-Scope; CONTEXT.md Deferred §). The mockups reuse the existing design-system geometry verbatim and vary **color only**.

**The real "visual contract" of this phase lives in the Color section.** It governs: (a) which semantic roles every scheme MUST fill, (b) the accessibility floor every scheme MUST meet, (c) the dual-ledger contrast requirement, and (d) the per-scheme exploration axes. The checker should weight Dimension 3 (Color) as the load-bearing dimension.

## Color

> **This is the load-bearing section.** Every one of the 4–5 schemes is "complete" only when it answers EVERY semantic role below for BOTH light and dark, meets the accessibility floor, and satisfies the dual-ledger contrast requirement. The current Wa-Modern palette (below) is the anchor being explored FROM, not a constraint to preserve except for the coral brand anchor (D-01).

### Current Palette (anchor / source of truth — `lib/core/theme/app_colors.dart`)

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) — background | `#FCFBF9` (warm ivory) | Page background, surfaces |
| Secondary (30%) — card / surface | `#FFFFFF` card, `#F5F4F2` muted | Cards, nested surfaces, dividers |
| Accent (10%) — primary coral | `#E85A4F` | FAB, primary CTA, active tab, input-active border, recording gradient |
| 日常 (Daily) ledger accent | `#5A9CC8` (blue) | Daily ledger amounts, tags |
| 悦己 (Joy) ledger accent | `#47B88A` (green) | Joy ledger amounts, tags |
| Olive (trends/ROI) | `#8A9178` | Positive-trend badges, ROI values |
| Shared (group mode) | `#D4845A` (orange) | Shared-ledger amounts, tags |

### Color Exploration Contract (D-01 → D-05)

| Decision | Locked rule for every scheme |
|----------|------------------------------|
| **D-01 Brand anchor** | Coral primary `#E85A4F` (or a recognizably coral hue) MUST stay present in all 4–5 schemes as a brand-memory point. Everything else — background, ledger accents, auxiliary, semantic — IS explorable (including cooler/more-neutral variants). Not a full open-ended "primary can change" exploration; not a conservative variations-only pass — the middle path. |
| **D-02 Dual-ledger contrast** | 日常 (Daily) and 悦己 (Joy) accents MUST read as a **clear visual contrast** (Daily = calm/cool/neutral; Joy = warm/bright) — NOT analogous/low-conflict like the current blue↔green. Tasteful, no garish clash. This is a hard pass/fail dimension for every scheme. |
| **D-03 No Joy celebration** | Joy is visually distinct/warmer but schemes MUST NOT introduce celebration-style affordances (glow, pulse, sparkle, milestone color-pop). Stays consistent with ADR-016 §5 100%-behavior contract (ambient state only, no discrete event). The existing `joy_celebration_overlay` is NOT a license to amplify. |
| **D-04 Warm-coexistence axis (UNLOCKED)** | Coral-primary (warm) vs Joy-accent (warm) tension is an **exploration axis**, not a fixed rule. The 4–5 schemes MUST span genuinely different resolutions, e.g.: (a) coral demoted to pure **action color** (FAB/CTA only, not a ledger semantic) + Joy in a separate warm family (gold/amber/orange); (b) Joy as a **coral-family bright tint** distinguished by lightness/saturation not hue; (c) other planner-devised resolutions. |
| **D-05 Daily-tone axis (UNLOCKED)** | "日常偏冷静中性" spans two ends each scheme set must cover: (a) continued cool blue/teal (continuity with `#5A9CC8`, cool↔warm contrast vs warm primary) OR (b) true neutral gray/slate (makes Joy's warmth pop hardest — "Daily = restrained background, Joy = highlight"). |

### Per-Scheme Semantic-Role Completeness Matrix (D-07)

A scheme is **incomplete** (and must be flagged by the checker) if any role below is unanswered for either light or dark. Role names match the already-renamed `AppColors` symbols (Phase 31) — schemes describe roles by these names, NOT a parallel naming.

| Group | Required roles per scheme | Light | Dark |
|-------|---------------------------|-------|------|
| Surface | background, card, backgroundMuted, backgroundSubtle, backgroundDivider | ✅ required | ✅ required |
| Text | textPrimary, textSecondary, textTertiary | ✅ required | ✅ required |
| Border | borderDefault, borderDivider, borderList, borderInputActive | ✅ required | ✅ required |
| Primary (coral anchor) | accentPrimary, accentPrimaryLight, accentPrimaryBorder, fabGradientStart, fabGradientEnd | ✅ required | ✅ required |
| 日常 Daily accent | daily, dailyLight (+ dark tag tint) | ✅ required | ✅ required |
| 悦己 Joy accent | joy, joyLight (+ dark tag tint) | ✅ required | ✅ required |
| Olive (trends/ROI) | olive, oliveLight, oliveBorder | ✅ required | ✅ required |
| Shared (group mode) | shared, sharedLight, sharedBorder, sharedChevron | ✅ required | ✅ required |
| Semantic | success, warning, error, info | ✅ required | ✅ required |

> Note: the existing palette has **no dedicated `success/warning/error/info` family** (it overloads olive/coral). Each scheme MUST now define these four semantic roles explicitly — this is net-new vs the current palette and feeds Phase 33's token system.

### Accessibility Floor (Claude's Discretion → WCAG default)

Apply to EVERY proposed palette. Flag any scheme role that cannot meet it.

| Pairing | Minimum contrast |
|---------|------------------|
| Body / label text on its surface | **≥ 4.5:1** (WCAG AA normal text) |
| Large text (≥18px bold / ≥24px) & UI affordances/borders | **≥ 3:1** (WCAG AA large/non-text) |
| Ledger accent amount text on card surface | **≥ 4.5:1** (amounts are read as data) |
| Coral CTA fill vs white icon/label | **≥ 3:1** (large affordance) |

Schemes must be implementable in Phase 33 without rework. A scheme with an un-fixable contrast failure on a load-bearing role is a selection-disqualifier and must be annotated as such in the synthesis.

### Selection Output (D-08 → ADR-018)

Selection allows **one scheme outright OR a named hybrid** (e.g. "Scheme B's Joy + Scheme D's Daily"). The selected palette + final hex for EVERY role in the matrix above is recorded in **ADR-018** (next sequential — current max ADR = 017; update `docs/arch/03-adr/ADR-000_INDEX.md`).

## Copywriting Contract

> Adapted: this phase has no end-user CTA/empty/error screens. The "copy" contract here governs the **scheme-presentation and selection artifacts** — the labels, rationale prose, and ADR copy the executor produces. This keeps the selection legible and the ADR auditable.

| Element | Copy contract |
|---------|---------------|
| Scheme labels | Each scheme has a stable, descriptive name (NOT "Scheme 1/2/3"). Name encodes its direction + warm-coexistence resolution, e.g. "Coral-Action + Amber-Joy", "Slate-Daily + Coral-Tint-Joy". Same label used in mockup frame title, synthesis doc, and ADR. |
| Per-scheme rationale | Each scheme's synthesis entry states: mood, mined-reference lineage (which brand DESIGN.md influenced it), its D-04 warm-coexistence resolution, its D-05 Daily-tone position, and any accessibility flags. |
| Mockup frame captions | Each of the 3 screens × 2 modes (light/dark) labeled unambiguously: `{scheme name} · {screen} · {light\|dark}`. Side-by-side layout so schemes compare directly at selection time. |
| Selection prompt | At the PALETTE-03 checkpoint, present all 4–5 schemes together (single final selection — no mid-exploration "narrow to 2" round, per CONTEXT discretion) and ask the user to pick one or name a hybrid. |
| ADR-018 copy | Title: "Palette Selection — v1.5". Records: selected scheme/hybrid name, the rejected alternatives + why, and a complete hex-per-role table (the matrix above) as the single authoritative reference for Phase 33. Status: 已接受 (Accepted) on user confirmation. |
| Destructive actions | **None.** This phase produces no code and no destructive operations. No confirmation flows required. |

## Hand-off Notes (for planner / executor)

- **No `lib/` changes this phase.** Hand-off to Phase 33 is ADR-018 (hex per semantic role) + the selected `.pen` scheme. CONTEXT D-01..D-08 + this contract are the governing inputs.
- **Pencil mechanics** (one `.pen` doc vs several, frame layout, side-by-side arrangement) are planner/executor's call — no `.pen` exists yet, create fresh.
- **Number of schemes:** target 4–5; at least 4 *genuinely distinct* directions required (ROADMAP Success Criterion #1). Planner decides 4 vs 5 based on how many distinct directions the mined references yield.
- **Reference mining (PALETTE-01):** VoltAgent/awesome-design-md brand DESIGN.md set (Linear, Notion, Stripe, Claude, etc.) — no local copy; fetch during research. No brand pre-biased (D-06) — weigh cool-minimal AND warm-neutral against the 家庭财务 + 双轨 + 和风 context.
- **Coral-as-action-only (D-04 resolution a) is already partly how the code uses coral** (FAB gradient, action gradient, input-active border, recording gradient) — lowers Phase-33 risk for schemes that lean that way.
- **Dark mockups are exploratory/forward-looking** — current `AppColorsDark` is profile-scoped only; full dark rollout is deferred (THEME-V2-02). Dark schemes are for evaluation, not a v1.5 ship commitment.
