    Sections:
    - 背景 (Context): The v1.7 multi-currency feature needs to store the exchange rate used at transaction save time. SQLite REAL (double) loses precision on 4-6 decimal place rates; re-multiplying a retrieved double against originalAmount can yield a different .round() result than the preview computed — causing a preview-vs-stored JPY divergence (reference Pitfall 1). All other currency-adjacent columns in the transactions table use TextColumn or IntColumn; no RealColumn precedent exists.
    - 考虑的方案 (Considered Options): (a) RealColumn (double) — simple arithmetic but precision loss risk; (b) TextColumn (string literal, full precision) — no precision loss, string comparison requires double.parse for arithmetic; (c) IntColumn scaled integer — extra complexity for no benefit
    - 决策 (Decision): TextColumn for appliedRate on transactions table (D-04). Drift field: TextColumn get appliedRate => text().nullable()(). Freezed field: String? appliedRate. Sync wire: string literal. Single parse site: inside convertToJpy() utility only.
    - 理由 (Rationale): Prevents preview-vs-stored divergence. Consistent with merchant/photoHash TextColumn pattern already in transactions. Rate string is audit-friendly (human-readable in DB inspection). ISO 4217 rate strings are short (max ~12 chars) — no storage cost concern.
    - 后果 (Consequences): All callers must use convertToJpy() (lib/shared/utils/currency_conversion.dart) — never inline arithmetic. Double comparison for >1% change check uses double.parse on both strings (D-05). appliedRate on exchange_rates cache table may use RealColumn (D-04 applies only to transactions.applied_rate, not the cache table).
    - 实施计划: Phase 40 — TextColumn in transactions_table.dart; currency_conversion.dart single parse site.

    Sections:
    - 背景: The transactions table gains three new nullable columns in v20→v21. HashChainService.calculateTransactionHash currently signs only four fields: transactionId, amount, timestamp, previousHash (confirmed by source inspection). Including new fields in the hash would invalidate ALL pre-migration chain hashes — every existing transaction would fail verifyChain.
    - 考虑的方案: (a) Include all three new fields in hash (maximum integrity) — invalidates all pre-v21 chains, requires re-sealing entire DB; (b) Exclude new fields from hash (backward-compatible) — existing chains stay valid; future currency-integrity is optional scope; (c) Version the hash formula — adds complexity, not needed at this scale
    - 决策: The three new nullable columns (original_currency, original_amount, applied_rate) are EXCLUDED from HashChainService.calculateTransactionHash. The hash formula remains: transactionId|amount|timestamp|previousHash. This is the sole definition of the hash scope for v21.
    - 理由: Hash chain exists primarily to detect tampering with the core financial record (amount + timestamp). Currency provenance fields are additive metadata. Backward compatibility with all existing (v1.0-v1.6) chains has greater value than marginal integrity gain from including rate metadata. The partial-triple domain invariant (CreateTransactionUseCase validation) is the tamper-detection mechanism for currency fields, not the hash.
    - 后果: Any architecture test for hash_chain_service.dart must assert the function signature does NOT include originalCurrency/originalAmount/appliedRate parameters. Future phases must not add currency fields to the hash without a new ADR.

    Sections:
    - 背景: Foreign-currency transactions have a complex edit model. The JPY amount is always derived from originalAmount × appliedRate (D-01 read-only contract). When editing, two inputs change in relation to one output. The edit-then-date-change scenario creates a policy conflict: RATE-06 says overrides are not clobbered, but Phase 41 SC-4 said unless the user changes date after overriding. Three distinct sub-decisions needed.
    - 考虑的方案: For D-01 (edit model): (a) full bidirectional three-field edit (any field recalcs others), (b) two-input one-output (original amount + rate → JPY, JPY read-only). For D-02 (override + date change): (a) silently preserve override, (b) silently clobber with new fetch, (c) dialog asking user. For D-03 (auto-recalc >1% change): (a) block save with confirmation, (b) silent recalc, (c) non-blocking toast with undo.
    - 决策 (three parts):
      D-01: JPY amount (amount column) is read-only in the edit UI for foreign-currency rows. Editable inputs are originalAmount and appliedRate only. JPY = (originalAmount / subunitToUnit × appliedRate).round() — always derived, never directly entered. Implements "原币是事实，日元是结果" principle.
      D-02: When a user has manually set appliedRate (manual override) AND subsequently changes the transaction date, the app shows a dialog asking: "保留手动汇率，还是按新日期重取？" — two choices, no silent default. The override flag is reset only if the user chooses "重取".
      D-03: When there is NO manual override and a date change triggers an auto-refetch whose new JPY amount differs from the old amount by more than 1%, the app recalculates silently but shows a non-blocking toast with an Undo action. Undo restores the previous rate. This is non-blocking (saving is not gated on confirmation). The 1% threshold compares JPY amounts (integer), not rates. The undo window is 5 seconds (Claude's discretion per CONTEXT.md).
    - 理由: D-01 prevents circular UI update loops and reflects the fundamental accounting invariant (original amount is the fact, JPY is presentation). D-02 respects the user's intentional override while not silently discarding new information. D-03 balances the never-block-save invariant with the user's right to know their JPY amount changed.
    - 后果: Phase 42 must implement the edit form per D-01 (two editable fields, one derived display). Phase 41 must expose a RateResult type that carries whether the rate was user-overridden, and the application use case must check override state before triggering the D-02 dialog signal. REQUIREMENTS.md DISP-04 wording "bidirectional linked editing" is superseded by D-01 — the canonical spec is this ADR.
    - 実施計画: Phase 42 (UI implementation); Phase 41 (rate override state management); Phase 40 (domain: partial-triple invariant covers the D-01 consistency guarantee at the data level).
    - Phase scope: Phase 42 (DISP-04), Phase 41 (RATE-06)
