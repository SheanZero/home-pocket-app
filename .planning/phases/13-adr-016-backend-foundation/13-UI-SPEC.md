## 4. `formatJoyCumulative` Display Format

This is the **only new visual deliverable in Phase 13**.

### Contract

| Property | Specification |
|----------|--------------|
| Type | Pure function `formatJoyCumulative(double rawSum, String currencyCode) â†’ String` |
| Rounding | Floor to integer before formatting (`rawSum.floor()`) |
| Decimal display | None â€” integers only |
| Unit suffix | None â€” cumulative Joy is a dimensionless count; no `/ÂĄ`, no `pt`, no unit label |
| Thousand separator | Locale-appropriate comma separator using `NumberFormat('#,##0', locale)` from `intl` 0.20.2 (already pinned in project) |
| Locale handling | `ja` â†’ comma separator (`1,234`); `zh` â†’ comma separator (`1,234`); `en` â†’ comma separator (`1,234`). All three locales use the same comma-based format. |
| Text style | `AppTextStyles.amountLarge` at the primary KPI tile position (HomeHeroCard line 478â€“480). If the value appears in a secondary/smaller context, use `AppTextStyles.amountMedium`. Both include `FontFeature.tabularFigures()` per CLAUDE.md mandate. |
| Zero case | `0` displays as `"0"` (no separator needed) |
| File location | `lib/infrastructure/i18n/formatters/joy_cumulative_formatter.dart` (co-located with deleted `joy_density_formatter.dart`) |

### `ptvfBaseFor` preservation

The `ptvfBaseFor(String currencyCode) â†’ double` function and the `_ptvfBaseByCurrency` map (JPYâ†’500.0, CNYâ†’25.0, USDâ†’5.0) are preserved **verbatim** in the new file. The recommendation use case (`GetMonthlyJoyTargetRecommendationUseCase`) depends on `ptvfBaseFor` at runtime.

### Worked example (2Ă—3 grid)

| Input `rawSum` | Locale / `currencyCode` | Displayed output |
|---------------|------------------------|-----------------|
| `1234.7` | `ja` / `JPY` | `1,234` |
| `1234.7` | `zh` / `CNY` | `1,234` |
| `1234.7` | `en` / `USD` | `1,234` |
| `50.0` | `ja` / `JPY` | `50` |
| `50.0` | `zh` / `CNY` | `50` |
| `50.0` | `en` / `USD` | `50` |

### Implementation note

Use `NumberFormat('#,##0', 'en_US').format(rawSum.floor())` as the implementation â€” the `#,##0` pattern produces comma-separated integers without a fractional part in all three supported locales. For `intl` 0.20.2, `'en_US'` locale produces correct comma separators; Japanese and Chinese locales also use comma-based grouping for this pattern. The planner verifies this against `intl` docs at implementation time.

## 5. `_SatisfactionHistogramOrFallback` Gate Rewire

### Problem

`_SatisfactionHistogramOrFallback` in `analytics_screen.dart` (lines 423â€“430) uses `dailyJoyPerYenProvider` solely as a sample-size gate: the widget shows a fallback message when the provider's `sampleSize < 5`. This is a compilation dependency on a provider that Phase 13 deletes. Without rewiring, the screen will not compile.

### Visual behavior contract: no change

The fallback message behavior is **visually identical** before and after rewiring:
- If fewer than 5 soul transactions exist for the selected period â†’ show fallback copy.
- If 5 or more soul transactions exist â†’ show the histogram.

The displayed fallback copy string (ARB-driven) does **not** change. Its ARB key is **not** renamed in Phase 13.

### Input switch

| Before (v1.1 baseline) | After (Phase 13) |
|------------------------|------------------|
| Gate reads `dailyJoyPerYenProvider`'s `sampleSize` field | Gate reads `happinessReportProvider`'s `totalSoulTx` field |
| `watch(dailyJoyPerYenProvider)` â†’ check `.sampleSize < 5` | `watch(happinessReportProvider)` â†’ check `.totalSoulTx < 5` |

`happinessReportProvider` is already consumed by `_KpiHero` elsewhere in `analytics_screen.dart`, so this is a new watch on an already-declared dependency â€” not a new provider import. The `totalSoulTx: int` field is on `HappinessReport` (populated by `GetHappinessReportUseCase`).

### Threshold constant

The `n < 5` threshold is unchanged. It remains the existing hard-coded bar value (aligned with the v1.1 5-bar histogram annotation). No ARB key is associated with this threshold number; it is a constant in code.
