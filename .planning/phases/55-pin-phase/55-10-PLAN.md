<objective>
Refactor the Settings security area per D-11 into the app-lock master toggle + revealed sub-items, build the reusable double-entry set-PIN flow (D-03), enforce re-authentication before disable/change (D-05), and wire the Phase 54 deep-link to auto-open set-PIN (D-10). This is the Settings-side control surface that arms/disarms the lock (LOCK-01/06).

Purpose: The user can enable/disable the lock and set/change the 4-digit PIN, never enabling lock without a PIN, never disabling/changing without re-auth.
Output: `SetPinScreen` + refactored `SecuritySection` + deep-link hook + tests.
</objective>

<artifacts>
## Artifacts this plan produces
- `SetPinScreen` (ConsumerStatefulWidget, in features/applock/presentation/screens): double-entry — enter 4 digits (`S.appLockSetPinTitle`) then re-enter to confirm (`S.appLockConfirmPinTitle`); mismatch -> `S.appLockPinMismatch` re-prompt + restart; on match -> `appLockService.setPin(pin)`; composes Plan 08 `PinKeypad`/`PinDots`; returns success via callback/Navigator result.
- Refactored `SecuritySection`:
  - app-lock master `SwitchListTile` (`S.securityAppLock`): ON -> push `SetPinScreen`; only on success `appLockService.enableLock()` (persist appLockEnabled=true) — else revert the toggle (never lock without a PIN). OFF -> `appLockService.reauth()` (biometric) or PIN re-entry; on success `appLockService.disableLock()`.
  - revealed when enabled: `生体認証で解除` sub-toggle (`biometricUnlockEnabled`, gated by `biometricAvailabilityProvider`) + `修改 PIN` entry (`S.securityChangePin`) -> reauth -> `SetPinScreen`.
  - existing `notifications` `SwitchListTile` unchanged.
- `settings_screen.dart`: when `scrollToSecurity` && lock-not-set, after the existing scroll, auto-open the set-PIN flow (D-10).
- Tests: `set_pin_screen_test.dart`, `security_section_test.dart`.
</artifacts>
