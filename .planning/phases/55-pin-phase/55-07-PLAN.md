<objective>
Create the application-layer `AppLockService` that centralizes the lock business logic so the gate, observer, lock screen, and Settings cannot diverge: the `lockEffective` predicate (D-01), set/verify PIN over the Plan 01 KDF, re-auth (D-05), and disable. Wire its Riverpod provider.

Purpose: One source of truth for "is the lock effective" and for all PIN operations (LOCK-01/06).
Output: `app_lock_service.dart` + `appLockServiceProvider` + unit tests.
</objective>

<artifacts>
## Artifacts this plan produces
- `AppLockService` (plain constructor-injected service in `lib/application/security/`, mirrors BiometricService injectability):
  - deps: `SettingsRepository`, `SecureStorageService`, `BiometricService` (for reauth).
  - `Future<bool> isLockEffective()` => `settings.appLockEnabled && (await getPinHash()) != null` (D-01 single source of truth).
  - `Future<void> setPin(String pin)` => `setPinHash(await derivePinPhc(pin))` (LOCK-06).
  - `Future<bool> verifyPin(String pin)` => parse stored PHC + `pin_kdf.verifyPin` (constant-time).
  - `Future<bool> reauth()` => biometric if `biometricUnlockEnabled && available` else false (caller falls back to PIN entry, D-05).
  - `Future<void> disableLock()` => `setAppLockEnabled(false)` + `deletePinHash()`.
  - `Future<void> enableLock()` => `setAppLockEnabled(true)` (only called AFTER setPin succeeds; Plan 10 enforces ordering).
- `appLockServiceProvider` added to `lib/infrastructure/security/providers.dart` (`@riverpod`, wires the three deps).
- `test/application/security/app_lock_service_test.dart`.
</artifacts>
