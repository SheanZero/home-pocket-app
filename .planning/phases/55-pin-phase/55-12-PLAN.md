<objective>
Close gap **G2** (55-UAT.md, UAT Test 1): app-lock authentication is satisfied by the
iOS **device passcode** instead of the app's OWN 4-digit Argon2id PIN, defeating the
central premise of the pin-phase.

Root cause (confirmed — .planning/debug/55-g2-auth-device-passcode.md):
`BiometricService.authenticate()` defaults `biometricOnly: false`, so `local_auth` uses
iOS `LAPolicy.deviceOwnerAuthentication`, which allows the device passcode as a fallback.
iOS renders its own passcode sheet and returns success, so a device-passcode unlock is
treated as a successful biometric unlock and the app never routes to its own PIN screen.
Both real call sites invoke `authenticate()` without `biometricOnly: true` and inherit the
passcode-allowing default:
  1. lib/features/applock/presentation/screens/app_lock_screen.dart:94 — `_runBiometric()`
     unlock auto-prompt (D-09; the path the user hit on-device).
  2. lib/application/security/app_lock_service.dart:67 — `reauth()` (D-05, before
     disable/change PIN).

The app's OWN PIN fallback surface is already fully built (`authenticate()` maps every
LocalAuthException → `AuthResult.fallbackToPIN()`; `_runBiometric()` stays on the Face ID
surface with 重試 / パスコードを使用 → app PIN page on any non-success). That flow is simply
bypassed whenever iOS is allowed to satisfy auth with the device passcode.

Chosen fix — **defensive default + explicit call sites** (the debug file's "Alternative
(defensive default)", keep-callers-explicit variant): flip
`BiometricService.authenticate`'s `biometricOnly` default to `true` (secure-by-default,
since this app never wants OS-passcode fallback) AND make both real call sites pass
`biometricOnly: true` explicitly. Justification: (a) secure-by-default guarantees no
future caller can silently reintroduce the device-passcode leak on this security-critical
path; (b) explicit args document the intent at the two lock surfaces and anchor the
regression assertions; (c) analyzer-safe — the project's flutter_lints ruleset does not
enable avoid_redundant_argument_values, so an explicit `true` equal to the default raises
no warning. Only two callers exist (grep-confirmed), both handled.

Follow TDD: RED (regression tests first) → GREEN (implement) → on-device human re-verify
of UAT Test 1 + Test 2 (Face ID hardware — cannot be automated).

Purpose: restore the LOCK-05/06/10 invariant — the app's own PIN is the credential, the
iOS device passcode is never offered or accepted.
Output: 3 source edits + 3 regression-test edits + an on-device re-verify checkpoint.
</objective>
