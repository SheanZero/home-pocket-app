# Phase 31: Terminology Rename - Context

**Gathered:** 2026-06-01
**Status:** Ready for planning

<domain>
## Phase Boundary

Unify the two ledgers' vocabulary across all three locales (zh/ja/en) **and** rename the internal identifiers that name them — taken to the **widest consistent scope**: ARB keys + values, all `AppColors` survival/soul symbols (including derived), the `LedgerType` domain enum (with a DB migration), and the `soul*/survival*` Dart file and class names.

**Locked locale mapping (from REQUIREMENTS.md):**

| Concept | zh | ja | en |
|---|---|---|---|
| Survival ledger | 日常 | 日常 (にちじょう) | Daily |
| Soul ledger | 悦己 | ときめき | Joy |

Internal identifiers are English-token based: `survival → daily`, `soul → joy` (ja value `ときめき` maps to the `joy` identifier).

**This phase does NOT change ledger logic/semantics** — only labels, identifiers, and the persisted enum string values.

</domain>

<decisions>
## Implementation Decisions

### Rename Scope Boundary
- **D-01:** Rename goes to the **widest ring** — all of: (a) ARB keys + values, (b) `AppColors.survival/.soul` + all derived color symbols, (c) the `LedgerType { survival, soul }` enum members, (d) the ~7 `soul*/survival*` Dart files and their class names. Nothing in the soul/survival vocabulary is left behind.
- **D-02:** **`LedgerType` enum is renamed** (`survival→daily`, `soul→joy`) **despite being persisted by string name** — this is a deliberate expansion past the milestone's original Out-of-Scope line. It requires a **Drift schema migration v17→v18** that: (1) `UPDATE`s existing `transactions.ledger_type` and `category_ledger_configs.ledger_type` rows `'survival'→'daily'`, `'soul'→'joy'`; (2) updates the `category_ledger_configs` `CHECK(ledger_type IN ('survival','soul'))` constraint to `IN ('daily','joy')` (SQLite requires a table-recreate migration for a CHECK change).
- **D-03:** **No P2P sync backward-compat layer.** App is pre-release (v0.1.0), no deployed older peers exist. Treated as a clean upgrade — `transaction_sync_mapper` simply emits/reads the new names; no dual-string acceptance path. (Revisit before any real release if old peers could exist.)
- **D-04:** **Hash chain is unaffected** — the per-transaction hash is `SHA-256(transactionId|amount|timestamp|previousHash)` (`hash_chain_service.dart:18`); `ledger_type` is NOT in the hashed payload, so rewriting stored values does not invalidate the chain. This is the key safety fact enabling D-02.
- **D-05:** File + class renames use **Serena `rename_symbol`** for atomic cross-file symbol renames and **`git mv`** for files (preserve history). E.g. `SoulVsSurvivalCard → DailyVsJoyCard`, `soul_vs_survival_card.dart`, `per_category_soul_breakdown*`, `get_soul_vs_survival_snapshot[_across_books]_use_case`, `get_per_category_soul_breakdown[_across_books]_use_case`, `soul_celebration_overlay`.
- **D-06:** **REQUIREMENTS.md Out-of-Scope must be amended** to remove/qualify the "Migrating database column names/values … not schema" line, since D-02 now performs a source+data enum migration.

### ARB Key Naming
- **D-07:** Keys are **redesigned semantically, not blind token-swap** — guiding principle is **concept-token substitution preserving structure** (`soul→joy`, `survival→daily`, keep prefix/case position and the rest of the key), with **targeted semantic corrections** where the mechanical name is inaccurate.
- **D-08:** Specific correction: `soulSatisfaction → joyFullness` (value is 充盈度/fullness, not "satisfaction"). This `satisfaction → fullness` correction propagates to color symbols too (see D-12).
- **D-09:** **Boundary:** apply to the ledger-vocab keys **and their sibling keys in the same key-group** (e.g. normalize `homeSurvivalExpense`/`homeSoulExpense` together, plus group neighbors for consistency) — but **NOT** a global retaxonomy of all ~533 keys.
- **D-10:** The **full old→new key mapping table is generated by researcher/planner and surfaced in PLAN.md** for visibility. Per-key approval is delegated to the D-07 principle (no per-key gate), but the complete table must be visible before execution.

### Derived Color Symbols
- **D-11:** **Rename ALL derived color symbols now** in Phase 31 (not deferred): `survivalLight→dailyLight`, `soulLight→joyLight`, `soulRoiBg→joyRoiBg`, `soulRoiBorder→joyRoiBorder`, `soulSatisfactionBg→joyFullnessBg`, `soulSatisfactionBorder→joyFullnessBorder`. `tagGreen` keeps its name (not old-vocab) but its definition `= soulLight` becomes `= joyLight`.
- **D-12:** **Phase 33 coordination seam:** Phase 33 (Color Token System) rebuilds the token system and consolidates duplicate constants. Phase 33 must treat these symbols as **already renamed** — it only *consolidates*, it does NOT re-rename. Note this explicitly so Phase 33 planning doesn't double-rename.

### ADR Strategy
- **D-13:** Create a **new ADR-017 "Terminology Unification v1.5"** (next sequential; current max is ADR-016). It satisfies TERMID-04's "ADR-015 or successor" via the successor path.
- **D-14:** ADR-017 records **all three**: (1) the canonical locale vocab mapping (the table above), (2) the identifier convention (D-07/D-08), and (3) the **`LedgerType` enum-rename-with-migration schema decision** including rationale (D-04 hash-chain-not-covered + D-03 pre-release-no-peers). Cross-link ADR-015.
- **D-15:** Append a one-line pointer to ADR-015 ("extended by ADR-017 …") per the arch append-only rule, and update `docs/arch/03-adr/ADR-000_INDEX.md`.

### Scope Expansions (added 2026-06-01, post-research)
- **D-16:** **`soul_satisfaction` is IN scope — full widest ring** (user decision, overriding research A1 recommendation). Rename the persisted DB column `transactions.soul_satisfaction → joy_fullness`, the Freezed field `Transaction.soulSatisfaction → joyFullness`, the sync-mapper JSON key `'soulSatisfaction' → 'joyFullness'`, and all ~50 dependent call sites (incl. `transactions_table.dart:35`, `transaction_sync_mapper`, analytics/use-cases). This requires a **second column migration** folded into the v17→v18 work (`transactions.soul_satisfaction` column rename — SQLite `ALTER TABLE … RENAME COLUMN`, data preserved). Per D-03 (pre-release, no peers) the sync-key change needs no backward-compat path. D-04 hash-chain safety also applies (soul_satisfaction not in the hashed payload — verify in research/plan).
- **D-17:** **Normalize already-acceptable non-literal phrasings to the literal vocab** (research A4): rewrite ja `暮らし`/`生活` and en `"Living Expenses"`-style values to the canonical `日常`/`Daily`. No grep gate depends on this; it's done for full vocab consistency per the D-07 spirit.
- **D-18:** **Rewrite ARB `@description` metadata too** (research A3): update description texts (e.g. "Soul ledger label" → "Joy ledger label") so the literal ROADMAP success-criterion #3 grep is unambiguously zero-hit even in its naive form, not just under a value-line filter.

### Claude's Discretion
- **Migration / commit sequencing** is delegated to the planner — big-bang vs staged (e.g. `LedgerType`+migration as its own commit, then ARB keys, then colors) is a planner call based on the dependency graph. **Hard constraint:** every step must keep the build green (`flutter analyze` 0 issues, `build_runner` clean-diff, tests pass).
- Test fixtures / golden handling: tests hardcoding `'survival'/'soul'` strings and any golden baselines affected will need updating to keep the suite green; the full golden re-baseline is Phase 34's job, but Phase 31 must not leave the suite red.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- **Serena `rename_symbol`** — atomic cross-file rename for the `LedgerType` members, AppColors symbols, and the soul*/survival* class names (updates imports/refs; far safer than Edit replace_all across 83+ enum sites).
- **Drift migration framework** — schema is at v17 (`entry_source` was the last bump in v1.2); v17→v18 migration slot is the home for the `ledger_type` data UPDATE + CHECK-constraint recreate.
- **`flutter gen-l10n`** — regenerates `S` localizations after ARB key renames; AUDIT-10 guardrail enforces clean generated-file diff.

### Established Patterns
- `LedgerType` is **persisted by `.name` string** (text column + `byName` + CHECK literal + sync payload) → renaming members is a data migration, not a pure source refactor.
- Hash chain hashes only `transactionId|amount|timestamp|previousHash` → ledger-type value rewrites are chain-safe.
- ARB has 533 keys/locale; only the ledger-vocab key-groups are in scope (D-09).
- AppColors gate (`grep AppColors.survival\|AppColors.soul`) does NOT catch derived symbols (`soulLight`, `soulRoiBg`, …) — D-11 renames them anyway for full consistency.

### Integration Points
- DB: `transactions.ledger_type`, `category_ledger_configs.ledger_type` (+ CHECK), schema version bump, `migrations` test.
- Generated: `S` localizations (gen-l10n), `.g.dart`/`.freezed.dart` regen after enum + ARB renames (build_runner).
- Phase 33 seam: derived color symbols renamed here; Phase 33 consolidates, does not re-rename (D-12).
- Phase 34 seam: golden re-baseline is Phase 34; Phase 31 only keeps the suite green.

</code_context>

<specifics>
## Specific Ideas

- `soulSatisfaction → joyFullness` (and `soulSatisfactionBg/Border → joyFullnessBg/Border`) — the "satisfaction"→"fullness" correction is intentional and should be applied uniformly to both ARB keys and color symbols.
- ja asymmetry is by design: the `joy` identifier maps to the ja value `ときめき` (coherent with the existing `ときめき指数` joy-index term), not a literal "joy" translation.
- Class rename examples the user expects: `SoulVsSurvivalCard → DailyVsJoyCard` (and matching file `git mv`).

</specifics>
