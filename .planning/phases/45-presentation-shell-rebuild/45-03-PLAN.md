<objective>
Create the typed card registry — the single source of truth for both render order and the `_refresh` invalidation union (D-B1). This file defines:

1. `AnalyticsCardContext` — the immutable snapshot carrying exactly `(bookId, startDate, endDate, trendAnchor, currencyCode, joyMetricVariant, isGroupMode, locale)`, the same keys the cards' `build` watches (D-A1/D-B2, no build/invalidation drift). (Plans 01/02's `*RefreshTargets` functions already bind to this type; if a minimal stub was created there, this plan fills it out — do not duplicate the class.)
2. `AnalyticsCardSpec` — `{ Widget Function(ctx) build; List<ProviderBase> Function(ctx) refreshTargets; bool Function(ctx) isVisible; String? sectionHeaderKey; }` with `isVisible` defaulting to always-true.
3. `analyticsCardRegistry` — a top-level ordered `List<AnalyticsCardSpec>` whose declaration order == today's render order, each `build` constructing the matching card (the 7 extracted cards + the 2 already-public leaf cards `DailyVsJoyCard` and `PerCategoryBreakdownCard`), each `refreshTargets` delegating to the per-card `<card>RefreshTargets(ctx)` from Plans 01/02, and the two family specs carrying `isVisible: (ctx) => ctx.isGroupMode` (D-B4).
4. `buildAnalyticsCardContext(BuildContext, WidgetRef, {required bookId})` — builds the ctx ONCE from the same providers the shell's `build` reads, so Plan 04's shell map and `_refresh` share one source.
5. `shellRefreshTargets(ctx)` → `[earliestTransactionMonthProvider(bookId: ctx.bookId)]` — the one non-card, shell-level refresh target (it's an analytics provider, so D-B3's union ⊆ analytics still holds).

Pick the spec-list over an abstract base class (RESEARCH Pattern 1): cards stay dumb `ConsumerWidget`s, the registry is trivially iterable by the Plan-05 D-B3 test without widget pumps, and the diff stays minimal (D-A1).

**D-C1/D-C2 honored by NON-ACTION:** the drill-host (a pushed GoRouter route — D-C1) is NOT built this phase and reserves NOTHING. Do NOT add any route, route param, donut-tap handler, or drill spec to the registry. The spec-list shape naturally accommodates "a future card whose tap pushes a route" with zero changes now; all drill implementation lands in Phase 46. This keeps the Phase 45 diff pure and goldens green (D-A1).

Purpose: Give Plan 04's shell a registry to map for `build` and to derive `_refresh` from, and give Plan 05's test an enumerable union.
Output: 1 new file `analytics_card_registry.dart`; no shell edits this plan.
</objective>
