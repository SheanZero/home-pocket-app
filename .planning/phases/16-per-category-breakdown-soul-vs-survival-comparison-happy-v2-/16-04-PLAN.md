<objective>
Extend the data layer with four new DAO methods + the matching repository interface + impl wiring, plus DAO-level unit tests. These methods become the source-of-truth queries for both Phase 16 surfaces:
- Per-category soul breakdown (single book + across books) — returns all categories sorted, no HAVING (Dart applies min-N per RESEARCH R1 to preserve low-N rows for the Other fold)
- Ledger snapshot (single book + across books) — returns per-ledger `(count, total spend)` tuple

Also adds the `_survivalExpenseFilter` constant for symmetry with the existing `_soulExpenseFilter` (anti-predicate-drift hygiene).

**Layer-boundary contract (revision iteration 1 — resolves checker BLOCKER 1 + 2):**
- DAO returns its own transient row tuples: `PerCategorySoulRowRaw` (renamed from `PerCategorySoulRow` to avoid collision with domain) and `LedgerSnapshotRow` (no domain collision; kept as-is).
- The **domain repository interface** (`lib/features/analytics/domain/repositories/analytics_repository.dart`) MUST return DOMAIN Freezed types from Plan 16-03 — `List<PerCategorySoulBreakdownItem>` for per-category methods. The interface MUST NOT import anything from `lib/data/`. CLAUDE.md Pitfall #2 (Domain → Data forbidden) is enforced by `import_guard` custom_lint and would fail CI.
- For `LedgerSnapshotRow`: it is a thin DAO-tier tuple (`ledgerType`, `totalAmount`, `entryCount`) with no domain Freezed counterpart in Plan 16-03 (the domain `SoulLedgerSnapshot` + `SurvivalLedgerSnapshot` are built by the use case, not the repository). To keep the domain interface Data-import-free, `LedgerSnapshotRow` is defined in BOTH the DAO (as a transient row class) AND mirrored in the domain models file as a re-exportable thin domain type — OR (preferred path) declared once in the domain models layer and reused by the DAO. RECOMMENDED CONCRETE PATH: add `LedgerSnapshotRow` as a `@freezed` (or plain Dart immutable class) inside `lib/features/analytics/domain/models/ledger_snapshot.dart` (already created in Plan 16-03 — the planner extends that file in this plan's Task 1 alongside the DAO additions). The DAO imports the DOMAIN type (Data → Domain is allowed) and constructs it directly. No mirror needed. Update Plan 16-03 acceptance criteria for `ledger_snapshot.dart` is unchanged because adding a new thin class to an existing file is in-scope of "ledger snapshot domain models".
- The **repository implementation** (`lib/data/repositories/analytics_repository_impl.dart`) lives in the Data layer, so it imports both the domain interface and the DAO. The impl performs the `PerCategorySoulRowRaw → PerCategorySoulBreakdownItem` conversion via `.map((r) => PerCategorySoulBreakdownItem(categoryId: r.categoryId, avgSatisfaction: r.avgSatisfaction, totalCount: r.totalCount)).toList()`. For `LedgerSnapshotRow` the impl passes the list through unchanged (same type produced by DAO and consumed by interface).

Purpose: Lock the data-fetch contract before Plan 05 (use cases) wraps it. Use cases must trust the repository to return correctly filtered, correctly sorted DOMAIN rows.
Output: Four new DAO methods + their row types, four new abstract repo methods returning domain types, four new @override impls with row-to-domain conversion, two new DAO test files covering query correctness + family-aggregate semantics.
