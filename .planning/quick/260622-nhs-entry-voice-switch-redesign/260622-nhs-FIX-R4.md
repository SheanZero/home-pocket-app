# Quick Task 260622-nhs · 修复 R4 — 语音会话生命周期 4 个真机问题

**日期:** 2026-06-22 · 前置 R3。真机发现 reset/状态/性能问题。文件：`voice_ptt_session_mixin.dart`、
`voice_listening_overlay.dart`(`VoiceRecordPanel`)、`manual_one_step_screen.dart`、语音服务
`application/voice/start_speech_recognition_use_case.dart` + `appSpeechRecognitionServiceProvider`。

## BUG A — 重置没清除「之前的录音」，需彻底清空 + 重新开始录音
**根因:** iOS 识别器在单次 `startListening(listenFor:30s)` 窗口内**累积** partial 文本（截图转写
「今天吃饭用了1450日元，今天吃饭用了1400」= 两句累积）。`_onVoiceReset → resetPttSessionState()` 只清 app 端
`_finalText/_partialText`，而 `restartPttListening()` 因 `isListening==true` **不重启** → 识别器内部 buffer 未清，
下一个 partial 又带出旧文本。
**改:** 重置 = **取消当前识别（`cancel()` 清识别器 buffer）→ 清空 `_finalText/_partialText/_parseResult/_mergedAmount` +
重建 merger → 重新 `startListening` 全新会话**。保证转写从空白开始、金额从快照值重新累积。不要用「仅当未在听才启动」的弱重启。

## BUG B — 重置后语音「假死」（不再响应）
**根因(假设):** reset 的 cancel→start 与 `onStatus` 的自动 re-arm **竞态**：`cancel()` 触发 `notListening`/`done`
→ `onStatus` 命中 `_continuousActive && _isRecording` → `_reArmPttListening()` 又 `startListening` → 与 reset 的
`startListening` **双重启动** → speech_to_text 插件卡死/静默。
**改:** 串行化重启：重置期间设 `_restarting=true` 守卫，使 `onStatus` 在重启窗口内**不**自动 re-arm；
`await cancel()` 完成后再 `await startListening()`（必要时极短延时让插件复位）；重启完成清守卫。
排查 `start_speech_recognition_use_case` / 服务层的 `cancel/stop/startListening/isListening` 语义，确保
不会在前一次未结束时重入。加测试覆盖「reset 后能再次产出结果并填表（无双启动、无卡死）」。

## BUG C — 状态需实时反映识别器真实状态（不要写死「正在聆听」）
**现状:** `VoiceRecordPanel` 标题硬编码 `listeningTitle`「正在聆听…」。
**改:** 引入会话状态枚举/字段（如 `PttListenStatus { listening, processing, stopped }`），由 `onStatus` +
「解析进行中」标志实时驱动：
- 正在聆听（recognizer listening）→ `listeningTitle`「正在聆听…」
- 正在解析（final 收到、`_fillFormFromText`/parse 进行中）→ 新 ARB「正在解析…」
- 停止聆听（done/notListening 且未 re-arm / 重启失败）→ 新 ARB「停止聆听」
面板标题 + 脉冲点颜色随状态更新（解析可用 amber，停止可用灰）。新增 ARB `voiceStatusProcessing`/`voiceStatusStopped` ×ja/zh/en + gen-l10n。host 把状态传入面板。

## BUG D — 解析慢
**根因:** ① `_onResult` final 分支**解析两遍**：`_parseFinalResult(text)`（含满意度）一遍，回调里 `_fillFormFromText(text)`
里 `parseUseCase.execute` **又解析一遍** → 冗余。② **只在 final（pauseFor 3s 静音后）才填表**；partial 只更新
`_parseResult` 不填 → 用户要等 ~3s 才见账目更新，感知很慢。
**改:**
- **去重解析:** 合并两条路径，复用同一个 parse 结果填表（`_fillFormFromText` 接收已解析的 data，或 `_parseFinalResult`
  直接驱动填表），final 路径只解析一次。
- **partial 实时填表:** 在已有 300ms partial debounce 解析里，**也驱动填表**（金额经 merger、分类/商家/日期实时更新），
  让账目随说随更新，而不是等 3s final。注意：partial 填表要幂等、可被 final 覆盖、可被重置回滚（快照仍以 startSession 时为基准）。
- JPY 路径不做汇率网络请求（现状已是，确认不回归）；外币 triple 仍仅在检测到外币时 fetch，不阻塞 JPY 主路径。
- 测量改善：说话到账目更新的延迟显著下降（从 ~3s+ 降到亚秒级 partial 更新）。
