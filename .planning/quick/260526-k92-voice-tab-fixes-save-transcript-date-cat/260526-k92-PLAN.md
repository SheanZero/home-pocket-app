### Item 2 — CRITICAL: 保存 button permanently gray on voice screen

**Current behavior.** `voice_input_screen.dart:112-113`:

```
bool get _canSave =>
    _hostCategory != null && _hostAmount > 0 && !_isSubmitting;
```

`_hostCategory` and `_hostAmount` are initialized to `null` / `0` (lines 107-108) and are ONLY written in `_stopRecordingAndCommit` (lines 343-346) — i.e. ONLY after a successful voice utterance.

**Root cause (CONFIRMED).** The voice screen never initializes a default category. Compare to `manual_one_step_screen.dart:122` which calls `_initializeDefaultCategory()` in `initState` and `manual_one_step_screen.dart:160-164` which sets `_selectedCategory` (and the parallel `_canSave` at line 89 flips to true). The voice screen omitted this entirely — the embedded `TransactionDetailsForm` has no category seeded, the host-cache mirror `_hostCategory` stays null forever, and `_canSave` never becomes true.

Secondary contributor: even if voice recognition fills the category, `_hostAmount` must also be > 0. Currently fine after a successful parse; but the bug surfaces at FIRST RENDER, not after parse.

**Target behavior.** On voice screen open, `_hostCategory` and `_hostAmount` defaults make 保存 enabled — either by:
- (a) Voice screen seeding defaults via the same `_initializeDefaultCategory()` flow as manual (preferred — keeps the two screens behaviorally aligned), OR
- (b) Relaxing `_canSave` to `_hostAmount > 0 && !_isSubmitting` (rejected: amount still defaults to 0, so this doesn't actually fix the bug at initial render; also loses category validation).

**Decision: (a).** Reuse the same default-category init pattern. Voice screen will load default L1 + first L2 via `categoryRepositoryProvider.findActive()` in `initState`, push them into the form via the existing `_formKey.currentState?.updateCategory(...)` (the form's setter already exists at line 214), and mirror to `_hostCategory`. `_hostAmount` stays 0 until voice/keypad fills it — but per upfront brief, 保存 should be enabled. So _canSave gates ONLY on `_hostCategory != null && !_isSubmitting` (drop the `_hostAmount > 0` clause — the form's `submit()` already validates amount > 0 and returns `validationError` which the existing snackbar surfaces).

This is also more user-friendly: tapping 保存 with no amount gives a clear "请输入金额" snackbar instead of silently doing nothing.

## Date phrase mapping table (Item 4)

| Phrase (zh) | Offset (days) | Notes |
|-------------|---------------|-------|
| 今天 / 今日 | 0 | Already in map |
| 昨天 / 昨日 | -1 | Already in map (昨日 via ja entry shares semantics) |
| 前天 | -2 | Already in map |
| 大前天 | -3 | **NEW** |
| 明天 | +1 | **NEW** |
| 后天 | +2 | **NEW** (parallel to 大前天) |

| Phrase (ja) | Offset (days) | Notes |
|-------------|---------------|-------|
| 今日 / きょう | 0 | Already in map |
| 昨日 / きのう | -1 | Already in map |
| 一昨日 / おととい | -2 | Already in map |
| 明日 / あした / あす | +1 | **NEW** (3 readings — colloquial) |
| 明後日 / あさって | +2 | **NEW** |

Multiple matches → take the one with the LARGEST `text.lastIndexOf(phrase)` (i.e. rightmost in utterance wins). This satisfies the brief's "if multiple date phrases appear in one utterance, take the LAST one" and the "contradictory phrases (e.g. 昨天今天)" handling.

Date is computed relative to `DateTime.now()` at parse time. Per brief: small distinction — "screen open time, not utterance time" — but parse runs in `_stopRecordingAndCommit` after release, which is utterance-end time. Difference is irrelevant in practice (sub-second). Use `DateTime.now()` as the parser already does (line 105).

## Risk callouts

- **Default category init race.** If `_formKey.currentState` is null when `_initializeDefaultCategory` resolves (form not built yet), the `updateCategory` call no-ops. Mitigation: wrap the `findActive` call in `WidgetsBinding.instance.addPostFrameCallback` so it runs AFTER first frame. The form's `updateCategory` has its own `if (!mounted) return` guard.
- **`updateDate` form contract.** Public method but used only by voice host. If a future host (OCR review screen) also wants date prefill, this becomes a 2nd consumer — that's fine, idempotent guard handles it.
- **Date LAST-wins reorder.** The change from "first match wins" to "rightmost-index wins" could theoretically affect existing voice utterances that happen to contain multiple date keywords. The corpus is small and the practical risk is low — but the existing zh/ja amount corpus tests will catch any aggregate regression. If a specific case breaks, add it back as an anchor case with the correct expectation per the new rule.
- **`_canSave` change.** Dropping `_hostAmount > 0` could let users tap 保存 with no amount. The form's `submit()` already returns `validationError` for amount=0, which the existing snackbar surfaces. Net: same user experience as today (no save), just a non-gray button + clear error message instead of silent dead button. Net win.
