<objective>
v1.3.1 ships **Option F (active learning from corrections)** from `.planning/research/voice-category-recognition-improvements.md` §3.F + §4. Closes the silent-orphan bug where the form-side write path and resolver-side read path use divergent keyword extractors (so learned rows are written under keys the resolver never looks up), and lets high-hitCount learned phrases participate in the substring fallback so `坐新干线` learned-then-uttered-as-`坐新干线去东京` resolves correctly without dict edits.

**Purpose:** Stop the whack-a-mole of hand-editing `default_synonyms.dart` for every new utterance phrasing. After this plan ships, the system learns from every user correction — within ~3 repeats of the same phrase, the resolver picks it up automatically.

**Output:**
- One canonical keyword string flows: resolver → VoiceParseResult.resolvedKeyword → form → recordCorrection → DAO → next-time resolver lookup.
- Learned rows (hitCount ≥ 3) join seeds in the substring fallback.
- A `dart run tool/dump_learned_keywords.dart` CLI to inspect what has been learned (no UI, no ARB strings).
- Worklog entry under `doc/worklog/`.

**Top-line cost answer for the orchestrator:** **NO Drift schema migration required.** The `category_keyword_preferences` table already has the `keyword` column wide enough to hold any user phrasing. The real gap is keyword-extraction divergence between two functions — fixed at the type layer (add a field to the freezed model + populate it once at the resolver) without touching the schema. Schema version stays at 17.
</objective>
