# Voice follow-up: amount-comma-loss, Shinkansen miscategory, save-always-clickable, transcript polish

> Follow-up to quick task **260526-k92** which the user device-tested and rejected with 5 issues. Single reproducer (Image #10):
> Utterance: `昨天做新干线用了12,450日元`
> Transcript correct; date correct (yesterday); **amount = 450 (wrong, lost the 12)**; **分类 = 交际费 > 聚会饮酒 (wrong; should be 交通 > 新干线)**; **保存按钮灰色不能点** (despite k92's seeded default).
>
> Five issues; ship as one logical change. No new packages, no new ML, no Phase 22 advisory backlog work.

## 3-line top-level diagnosis (for Issues 1, 2, 5)

1. **Issue 1 (12,450 → 450):** `日元` is NOT a recognized currency suffix (`voice_currency_suffixes.dart:22-31`), so the comma-aware regex `\d{1,3}(?:,\d{3})*` at `voice_text_parser.dart:76-79` is bypassed; the fallback standalone `\d{3,7}` regex at line 81 matches only `450` after the comma split the digit run.
2. **Issue 2 (新干线 → 交际费>聚会饮酒):** `新干线` / `新幹線` are NOT in `default_synonyms.dart` (only generic `交通費` / `定期` / `地铁` / `公交` / `打车` / `タクシー` are seeded). Resolver returns null → in `_stopRecordingAndCommit` the unconditional `setState(() => _hostCategory = category)` at `voice_input_screen.dart:392-395` then **overwrites the seeded default with null** — but the form's `state.updateCategory()` was guarded by `if (category != null)`, so the FORM keeps showing the seeded `cat_food_groceries` until... wait, see Issue 5 below for the actual surface bug. The 交际费>聚会饮酒 result the user sees is residue from a prior `recordCorrection` write that pinned a long-keyword → `cat_social_drinks` in `category_keyword_preferences` (user must have manually re-categorized once in a prior test).
3. **Issue 5 (保存 button still gray):** `_stopRecordingAndCommit` runs `setState(() { _hostAmount = amount; _hostCategory = category; })` unconditionally at `voice_input_screen.dart:392-395`. When voice resolver finds no category (Issue 2 scenario), `category` is `null` and this **nukes the seeded default `_hostCategory`**, flipping `_canSave` to `false` → save button renders at alpha 0.4 ("gray").  Per user's mental model, the fix is "save always clickable on voice; validation surfaces on submit" — drop the category gate entirely on voice and let the form's existing `submit()` surface the `pleaseSelectCategory` snackbar.

## Per-item diagnosis + fix

### Issue 1 — CRITICAL: Number parser drops the thousands group (`12,450日元 → 450`)

**Code path (concrete lines).** `VoiceInputScreen._stopRecordingAndCommit` (line 334) → `parseVoiceInputUseCaseProvider.execute(text, localeId)` (line 350) → `_textParser.extractAmount(text, localeId: 'zh-CN')` (`parse_voice_input_use_case.dart:50`) → `voice_text_parser.dart:45 extractAmount`. The hint pattern at line 31 detects no kanji numeral in `12,450日元` (no `一二三...千百万` characters, only Arabic digits + commas + `日` + `元`), so the function takes the Arabic branch at line 55 → `_extractArabicAmount(text)` at line 69.

`_extractArabicAmount` tries 3 regexes in order (lines 70-82):
- `RegExp(r'[¥￥]\s*(\d{1,3}(?:,\d{3})*(?:\.\d{1,2})?)')` — needs leading `¥` or `￥`; no match.
- `RegExp(r'(?<!\d)(\d{1,3}(?:,\d{3})*(?:\.\d{1,2})?|\d{4,7}(?:\.\d{1,2})?)\s*(?:円|えん|yen|元|块|塊)')` — the trailing `(?:円|えん|yen|元|块|塊)` does NOT include `日元`. The naked `元` could match the `元` in `日元`, but then the lookbehind has to anchor to a digit run. Crucially the digit `(\d{1,3}(?:,\d{3})*...)` would still need `12,450` followed directly by the suffix. In `12,450日元`, the `日` between `450` and `元` defeats this match (regex expects digit immediately before `元`). So this pattern misses.
- `RegExp(r'(?<!\d)(\d{3,7}(?:\.\d{1,2})?)(?!\d)')` — standalone digit run. `12` is only 2 digits → fails `\d{3,7}`. `,` is not a digit so it doesn't satisfy `\d`. `450` is preceded by `,` (not `\d`, so `(?<!\d)` holds) and followed by `日` (not `\d`, so `(?!\d)` holds) → **MATCHES `450`**. `extractAmount` returns 450.

**Confirms Hypothesis A** from the brief: parser splits on commas / non-digit characters and ends up with the trailing 3-digit run. Hypothesis B is the same root. Hypothesis C (chunk merger split) is **ruled out** — even if the merger had not yet committed, the screen calls `parseVoiceInputUseCaseProvider.execute(_finalText)` directly with the WHOLE `_finalText` at line 350 (`text = _finalText.isNotEmpty ? _finalText : _partialText`), and the merger's amount only wins via the separate `_mergedAmount ?? data.amount ?? 0` at line 371 (and the merger sees the same final text via `feedChunk` at line 510).

**Fix.**

1. **Add `日元` to `VoiceCurrencySuffixes.all`** (`lib/shared/constants/voice_currency_suffixes.dart:22`). Insert at index 0 (longest-first ordering — two-char `日元` must beat one-char `元`):
   - New list ordering (top to bottom): `'日元'`, `'块钱'`, `'えん'`, `'yen'`, `'円'`, `'元'`, `'块'`, `'塊'`, `'ドル'`.
   - This fixes `_extractKeyword` in `parse_voice_input_use_case.dart:124-131` AND `extractAndMatchMerchant` in `voice_text_parser.dart:482-491` for free — both call `VoiceCurrencySuffixes.regexAlternation`.

2. **Add a comma-tolerant arabic pattern that handles the suffix in `_extractArabicAmount`.** Replace the second pattern in the list at `voice_text_parser.dart:76-79` with one that uses the centralized suffix set AND handles full-width comma `，` AND tolerates intermediate non-digit before the currency word:
   - Build the suffix alternation inline from `VoiceCurrencySuffixes.regexAlternation` (import `voice_currency_suffixes.dart`).
   - The new regex: `RegExp(r'(?<!\d)(\d{1,3}(?:[,，]\d{3})*(?:\.\d{1,2})?|\d{4,9}(?:\.\d{1,2})?)\s*(?:' + VoiceCurrencySuffixes.regexAlternation + r')', caseSensitive: false)`.
   - Normalize the captured group by stripping BOTH `,` and `，` before `double.tryParse` (line 87 → `replaceAll(RegExp(r'[,，]'), '')`).
   - Bump the upper digit limit from `\d{4,7}` to `\d{4,9}` so 8-9-digit amounts (rare but technically valid `1234567890` etc) don't silently drop. Keep the `amount < 10000000` guard at line 89 — that's a separate sanity check.

3. **Also add a comma-aware standalone pattern as the last fallback** (in case the suffix isn't recognized for some other reason): `RegExp(r'(?<!\d)(\d{1,3}(?:[,，]\d{3})+)(?!\d)')` — matches `12,450` without any currency suffix, returns 12450. Insert as the THIRD pattern (before the existing 3-7-digit fallback). The existing 3-7-digit fallback stays as the last-resort but now only fires when no commas are present at all.

**Why this doesn't break spelled-out numbers.** The Arabic path is gated by `!hasNumeralHint` at line 52. Any utterance with kanji numerals (`一万二千四百五十`) routes to the state machine instead — `_runStateMachine` is unchanged. The fix is surgical to the Arabic path only.

**Why this doesn't break the existing `480日元 → 480` case.** `480日元` has no comma; the new comma-aware regex won't match (no `,\d{3}` group), but the standalone fallback `\d{3,9}` still matches `480` and the suffix `日元` is now recognized by `_extractKeyword` for keyword stripping. Equally important — `\d{1,3}` in the new regex matches `480` alone with the suffix `(?:...|日元|...)` and returns 480. Pattern order: comma+suffix → bare comma fallback → standalone-with-suffix → standalone. All four paths converge on the correct integer.

   **Pragmatic mitigation:** add `_extractKeyword` substring scanning AFTER the exact-match step 2 misses.    **However:** introducing substring scanning into the resolver opens false-positive risk (e.g. a learned `本` seed would match every utterance containing `本`).    - On miss → call NEW `_preferenceRepository.findSeedRowsByKeywordSubstring(keyword)` → DAO query: `WHERE hit_count = 0 AND ? LIKE '%' || keyword || '%' ORDER BY length(keyword) DESC LIMIT 1`. Drift syntax: use `customSelect` with that SQL.
   - On match → normalize via `_ensureL2`, return with `MatchSource.keyword`, confidence 0.80 (lower than exact-match's 0.85 — substring is weaker signal).

   **Files touched for this mitigation:**
   - `lib/data/daos/category_keyword_preference_dao.dart` — add `findSeedRowsByKeywordSubstring(String text)` method.
   - `lib/data/repositories/category_keyword_preference_repository_impl.dart` — wire it through.
   - `lib/features/accounting/domain/repositories/category_keyword_preference_repository.dart` — add the abstract method.
   - `lib/application/voice/voice_category_resolver.dart` — call it after the exact-match step 2 miss.

   **Alternative (lighter)** if the substring DAO method is too invasive for a quick task: do the substring scan IN-MEMORY inside the resolver. Read all seed rows once at first call (cache in the resolver instance), then scan in Dart. ~150 seeds, trivial cost. Trade-off: resolver becomes stateful and needs the seed cache invalidated on... nothing (seeds don't change at runtime). Acceptable for a quick task. **Pick the in-memory option** — saves touching the DAO + repo interface.

   **In-memory implementation:** add private `List<({String keyword, String categoryId})>? _seedCache` to `VoiceCategoryResolver`. Lazy-load on first `resolve()` call via `_preferenceRepository.findAllSeedRows()` (NEW method — fetch where `hitCount = 0`). After exact-match step 2 misses, scan the cache for the longest seed key contained in the input keyword.

   **Files touched (in-memory variant):**
   - `lib/features/accounting/domain/repositories/category_keyword_preference_repository.dart` — add abstract `Future<List<CategoryKeywordPreference>> findAllSeedRows()`.
   - `lib/data/repositories/category_keyword_preference_repository_impl.dart` — implement with `findBy(hitCount: 0)`-equivalent.
   - `lib/data/daos/category_keyword_preference_dao.dart` — add `findAllSeeds()` returning `WHERE hit_count = 0`.
   - `lib/application/voice/voice_category_resolver.dart` — cache + substring scan.

6. **Confidence note for the substring scan.** A substring hit `新干线` inside `昨天做新干线用` confidently maps to transportation. False positives mostly come from very short seed keys (1-2 char). Mitigation: **skip seeds with key length < 2** in the substring scan. ## Risk callouts

- **Resolver substring scan false positives.** Mitigated by `length >= 2` filter + longest-key-wins sort. The single-char Chinese seeds (`本`, `服`, `药`, `书`) and Japanese (`服`, `靴`, `薬`) are explicitly EXCLUDED from substring scan — they still serve via exact match. Watch for: if a new utterance contains a 2-char seed substring that the user didn't mean (e.g. `飞机` inside the merchant name `飞机楼咖啡店`), the resolver could miscategorize. Manual correction + `recordCorrection` upsert is the recovery path — same as today's behavior for other ambiguous matches.

- **Seed cache memory.** `_seedCache` holds all `hitCount=0` rows in the resolver instance. ~150 seeds × ~50 bytes/row ≈ 7.5 KB. Negligible. Cache lives for the resolver's lifetime (one instance per app, since the provider is keepAlive by default). Invalidation: never needed at runtime; seeds are immutable per app version.

- **Comma regex precedence.** The new pattern list order in `_extractArabicAmount` is: `¥xxx` → `xxx,xxx + suffix` → `xxx,xxx standalone` → `xxxx-xxxxxxxxx + suffix` → `xxx standalone`. Each is tried in order and the first match wins. The 7-digit cap bump to 9 covers million+ amounts that were previously silently dropping. Verified `1,234,567日元` → 1234567 in the corpus.

- **Issue 3's UX trade-off.** Voice tab save button is now ALWAYS clickable, which means users may tap save before any voice input. They get a `pleaseSelectCategory` snackbar — same outcome as today (no save) but a visible toast instead of a silently dead button. Per user's explicit request.

- **Polluted preferences from prior testing.** Some users (the developer running k92's tests) may have a `category_keyword_preferences` row that exact-matches a long utterance text → `cat_social_drinks`. This row has `hitCount >= 1` and will WIN over the new `新干线` seed (hitCount=0). The fix does NOT auto-purge. User must manually re-correct one utterance to bump the correct mapping above the polluted one. Documented in the checkpoint.

- **Existing date corpus tests** (`voice_date_corpus_test.dart`) iterate `voiceDateCorpusZh` / `voiceDateCorpusJa` — adding the new `voiceCategoryCorpusZh` / `voiceCategoryCorpusJa` lists to the SAME fixture files is safe (different typedef, different test file consumes them). Verified the existing date test only references `voiceDateCorpusZh` / `voiceDateCorpusJa` (lines 25/46 of `voice_date_corpus_test.dart`).
