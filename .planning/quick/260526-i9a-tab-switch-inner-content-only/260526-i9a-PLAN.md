<objective>
Fix the jarring full-screen transition when switching between the three input-mode tabs
(ж‰‹еЉЁиѕ“е…Ґ / OCR / иЇ­йџі) on the Add-Transaction flow.

Purpose: The current implementation does a `Navigator.pushReplacement(MaterialPageRoute(...))`
on every tab tap, which slides the whole screen вЂ” including the AppBar and the tab strip
itself вЂ” across the viewport. The user sees the tab they just tapped slide off the screen
and a "new" identical-looking tab slide in. Visually it reads as "broken" even though the
target screen is correct.

Output: A modified `navigateToEntryMode()` that swaps the route with **no transition
animation**, so the user perceives only the body below the tab strip changing.
</objective>

## Current behavior

The Add-Transaction flow is implemented as **three separate `Scaffold` screens**:

- `lib/features/accounting/presentation/screens/manual_one_step_screen.dart` (`ManualOneStepScreen`)
- `lib/features/accounting/presentation/screens/ocr_scanner_screen.dart` (`OcrScannerScreen`)
- `lib/features/accounting/presentation/screens/voice_input_screen.dart` (`VoiceInputScreen`)

Each screen builds its own `AppBar` (with `Icons.close`, title `l10n.addTransaction`,
centered) and its own copy of `EntryModeSwitcher` (the InputModeTabs strip) at the top of
its body. The three screens differ only in their body content below the tab strip.

When the user taps a different tab, `InputModeTabs` calls back into
`EntryModeSwitcher`, which calls `navigateToEntryMode()` in
`lib/features/accounting/presentation/navigation/entry_mode_navigation_config.dart`. That
function builds a `MaterialPageRoute<void>` and calls `Navigator.of(context).pushReplacement(route)`.

`MaterialPageRoute` uses the platform-default page transition (right-to-left slide on
iOS, vertical slide + fade on Android). Because the AppBar and the tab strip are
**inside** each Scaffold, they slide off-screen along with the body. The user sees the
entire screen transition even though structurally the AppBar and tab strip are visually
identical on all three screens вЂ” hence "ж•ґдЅ“йЎµйќўе€‡жЌўпјЊж•€жћњйќћеёёе·®".

## Target behavior

After the fix, tapping a different tab should look as if **only the area below the tab
strip swaps**. The AppBar and the tab strip stay visually stationary. The pill-highlight
animation inside `InputModeTabs` (the existing `AnimatedContainer`, 200ms) still plays as
the selected tab pill slides to the newly tapped tab. Everything else (back/close
behavior, deep linking, route stack semantics) is unchanged.

## Approach decision

Two options were considered:

1. **Restructure into a single host screen with `IndexedStack`/`AnimatedSwitcher` over
   the body region.** This is architecturally cleaner but requires extracting the AppBar
   into a host, threading state out of each Scaffold, and rewiring the existing entry
   points (`MainShellScreen` FAB, ocr_review_screen, transaction_edit_screen, etc.).
   That's a refactor вЂ” **out of scope** for a "quick" UI fix per the constraints.

2. **Suppress the route transition animation in `navigateToEntryMode()`.** Replace the
   `MaterialPageRoute` with a `PageRouteBuilder` that has `transitionDuration:
   Duration.zero` and a `transitionsBuilder` that returns the child unmodified. Because
   the three Scaffolds already render identical AppBars and identical tab strips at the
   same position, removing the slide animation makes the swap visually indistinguishable
   from an in-place body swap вЂ” exactly what the user asked for.

**We chose option 2** вЂ” minimal change (single file, ~15 lines), zero risk of breaking
the per-screen state machines (e.g. `ManualOneStepScreen`'s focus/keypad logic,
`VoiceInputScreen`'s locale-readiness mixin and event handler mixin), zero impact on
existing tests.
