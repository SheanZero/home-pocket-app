### Item 3 — CRITICAL BUG: KeyboardToolbar's 记录 button doesn't save (manual tab + IME open)

**Current:**
1. User taps 商家 or 备注 TextField → `_merchantFocus`/`_noteFocus` gains focus → `_handleFocusChange` fires → `setState(_isTextFieldFocused = true)`.
2. Build runs: `if (_isTextFieldFocused) Positioned(... child: KeyboardToolbar(onSave: _trySave, ...))` mounts the floating toolbar at the bottom-of-IME position.
3. User taps the toolbar's 记录 button.
4. **The TextField's `onTapOutside` fires FIRST** (because Flutter's `TapRegion` semantics route the pointer-down event to the TextField's region detector before the InkWell's `onTap` resolves on pointer-up). `onTapOutside` calls `FocusScope.of(context).unfocus()`.
5. The focus loss propagates: `_merchantFocus.hasFocus = false` → `_handleFocusChange` listener → `setState(_isTextFieldFocused = false)`.
6. In the next build, the `if (_isTextFieldFocused)` branch evaluates false → `Positioned(..., KeyboardToolbar)` is no longer in the tree.
7. The pending pointer-up event reaches a disposed/unmounted InkWell → `onSave` is never invoked.
8. User-visible result: IME dismisses, screen stays open, no save snackbar, no navigation. Matches "returns to input" report exactly.

**Why SmartKeyboard's Save key works correctly:** SmartKeyboard is only mounted when `_showSmartKeypad == true` (i.e. `!_isTextFieldFocused`). It is NEVER on-screen at the same time as a focused TextField — so `onTapOutside` is not in play.

**Why this regression slipped through:** No existing test taps the toolbar's record button to assert a save actually happens. The Phase 19-W1 test at `manual_one_step_screen_test.dart:366` only checks the toolbar's `isSubmitting` flag (visual disable state), not the InkWell's tap → save path.

**Target:** Tapping the toolbar's 记录 button while a TextField is focused MUST:
- Invoke `_trySave` (which calls `_save` when `_canSave`)
- NOT trigger `onTapOutside` on the TextField
- Preserve all other `onTapOutside` behavior (taps anywhere ELSE — e.g. on the form scrollview, on the AmountDisplay, on the AppBar — still unfocus the TextField as today)

**RootCause:** `onTapOutside: (_) => FocusScope.of(context).unfocus()` on the TextFields (`transaction_details_form.dart:610` and `:696`) does not exclude the KeyboardToolbar from its "outside" definition. The fix is Flutter's first-class mechanism for exactly this pattern: `TapRegion` with a shared `groupId`.

**Fix specification:**

1. Define a shared constant (top-of-file in `keyboard_toolbar.dart`):
   ```
   const String kKeyboardToolbarTapRegionGroup = 'manual-entry-keyboard-toolbar';
   ```

2. In `keyboard_toolbar.dart` `build()`, wrap the existing `Material(...)` outer in a `TapRegion`:
   ```
   return TapRegion(
     groupId: kKeyboardToolbarTapRegionGroup,
     child: Material(...existing...),
   );
   ```
   (Or wrap at the manual_one_step_screen.dart mount site if the toolbar is reused elsewhere — currently it's only used by manual_one_step_screen.dart, so wrapping inside `KeyboardToolbar.build()` itself is cleaner and keeps the wiring co-located with the widget.)

3. In `transaction_details_form.dart`, set `TextField.groupId` to the same constant on BOTH the merchant TextField (line 586-626) and the note TextField (line 673-712):
   ```
   TextField(
     ...,
     groupId: kKeyboardToolbarTapRegionGroup,  // <-- ADD
     ...
   )
   ```
   Import the constant from `keyboard_toolbar.dart`.

   **Why this is the correct fix per Flutter docs:** From `TextField.groupId` (Flutter API): "TapRegions with the same groupId share their tap state — taps inside any region in the group don't count as taps outside any other region in the group." This is the canonical fix for IME-accessory-toolbar patterns and is used by Flutter's own `Cupertino*` widgets.

4. Verify: with `groupId` matched, `onTapOutside` on the TextField will NOT fire for taps on the toolbar — the InkWell's `onTap` runs as expected → `_trySave` fires → save proceeds.

**Why not "make `_trySave` work even after unfocus":** That's a workaround, not a fix. The root cause is the spurious unfocus event being emitted at all. Removing it (via `TapRegion`) is cleaner, matches Flutter's first-class API, and prevents future regressions where some unrelated state piggybacks on `_isTextFieldFocused`.

**Why not "delay unfocus by one frame":** Race-condition-shaped fix. Brittle. Rejected.
