<objective>
Fix the bug where deleting all data (or importing a backup) in Settings leaves the Home / List / Analytics tabs showing stale data until the app is restarted.

Purpose: `ClearAllDataUseCase` wipes the books/categories/transactions tables but (a) never recreates a default book, (b) never invalidates any Riverpod provider, and (c) cannot fix the active book id because `bookId` is NOT a provider — it is captured once at boot in `_HomePocketAppState._initialize()` and threaded down as a constructor parameter to every screen. After a wipe the captured `_bookId` dangles at a deleted row; the IndexedStack tab shell keeps all four tabs permanently mounted, so their one-shot FutureProviders never re-fetch. `_importBackup` has the identical hidden bug (its `_restoreData` deletes all books then inserts the backup's books, whose ids differ from the captured `_bookId`).

Output: A centralized, DRY refresh mechanism. Both destructive Settings actions fire a single global reset signal; the app root re-bootstraps a fresh default book, recomputes the active bookId, invalidates every data-provider family, and rebuilds the shell — all without an app restart.
</objective>

<design_decision>
## Why an app-root re-bootstrap + centralized invalidation (not per-widget invalidate)

Two INDEPENDENT failure modes exist; both must be fixed:

1. **Dangling/stale captured bookId.** `bookId` is a plain `String` constructor parameter (NO `currentBookIdProvider` exists in this codebase — grep confirms). Pure provider invalidation inside `DataManagementSection` CANNOT fix this: every screen would simply re-query providers keyed by the same now-deleted `bookId`. The active book id can only be recomputed where it is owned — `_HomePocketAppState` — by re-running the bootstrap and calling `setState`. `EnsureDefaultBookUseCase` mints a fresh `Ulid()` id, so after a clear the new bookId differs from the captured one; rebuilding `MainShellScreen(bookId: newId)` re-keys every bookId-family provider to a brand-new instance that fetches fresh.

2. **Provider caches not released.** For the import-the-same-backup edge (restored book keeps its original id, so new bookId == old bookId → no re-key) and for the providers that are NOT keyed by bookId (`shadowBooksProvider`, `appSettingsProvider`, `currentLocaleProvider`), re-keying is not enough. These need explicit whole-family `ref.invalidate(family)`.

**Chosen mechanism (covers both, DRY):**
- A global `DataResetSignal` notifier (`dataResetSignalProvider`). Both Settings success paths call `.fire()` — ONE line each.
- `_HomePocketAppState` uses `ref.listen(dataResetSignalProvider, ...)` (side-effect → `ref.listen`, per CLAUDE.md, never `ref.watch`) to run ONE shared routine: re-seed (idempotent, count-guarded — restores categories after a wipe, no-op after import) + `ensureDefaultBookUseCase` → new bookId, then `invalidateAllDataProviders(ref)`, then `setState(_bookId = newId)`.
- `invalidateAllDataProviders` invalidates whole families (no keys needed — `ref.invalidate(family)` discards every keyed instance), the same plain-top-level-function pattern as the existing `invalidateTransactionDependents`.

This funnels clear-all AND import through a single routine and reuses the established refresh idiom.
</design_decision>
