<objective>
Author the ONE genuinely-new pure bit of Phase 44: a shared L1-rollup helper with a LOCKED public surface that rolls category data up to the L1 parent. It is the lynchpin of OVW-01 (D-10/D-11) — the donut's "10 L1 categories descending" display transform — AND the single source-of-truth the Plan 03 drill-down summary reuses for its subtotal/count (D-11, avoids subtotal drift per RESEARCH Pitfall 3).

The interface is deliberately fixed (not planner-discretion) so the same-phase Plan 03 executor binds to a stable API and CANNOT invent a parallel aggregation: the helper exposes ONE shared `l1AncestorOf` rule plus TWO rollup entrypoints over the SAME rule — `rollupCategoryBreakdownsToL1(...)` for the donut (consumes `MonthlyReport.categoryBreakdowns`) and `l1RollupFromTransactions(...)` for the drill (consumes the raw `List<Transaction>` the drill use case already has). Because both go through `l1AncestorOf`, the donut slice and the drill header are mathematically the same number.

Purpose: OVW-01 is locked as zero-new-data-work (D-10) except this one shared pure transform (D-11). Authoring it once, in domain-pure code, lets both the donut and the drill summary consume the same ancestor rule — no second rollup, no drift.
Output: `lib/features/analytics/domain/category_l1_rollup.dart` (pure, no Flutter import) + its unit test. No use case, no DAO, no provider, no migration.
</objective>
