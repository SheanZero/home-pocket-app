<objective>
Adopt the D-14 spillover in the two Phase 18 hosts that previously relied on the form widget's now-deleted internal amount-editing sheet: `TransactionEditScreen` and `OcrReviewScreen`. Each host now renders its own `AmountDisplay` above the embedded `TransactionDetailsForm`, and on tap opens the new `AmountEditBottomSheet` (extracted by Plan 01 Task 2) which pushes the confirmed integer back to the form via `_formKey.currentState!.updateAmount(int)`.

Purpose: Plan 01 Task 3 removed `_editAmount()` and the amount DetailInfoRow from the form widget. Without this plan, the edit screen and OCR review screen lose their amount-editing affordance entirely — that would be a Phase 18 regression. This plan restores the affordance by having each host own its AmountDisplay and call the new shared sheet widget. Neither host adopts the persistent-keypad pattern (D-14 invariant: "Only ManualOneStepScreen uses the persistent SmartKeyboard pattern" — the other hosts keep tap-amount-to-open-sheet for low-volume edit use cases).

Output: Two modified host screens (each gains an `AmountDisplay` + a tap handler that calls `AmountEditBottomSheet.show`) + two small widget tests proving the AmountDisplay+sheet integration round-trip works.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Host display state ↔ form internal state | Both `transaction_edit_screen.dart` and `ocr_review_screen.dart` maintain a local `_displayAmount` int that must stay in sync with the form's internal `_amount` via `_formKey.currentState?.updateAmount(...)`.
