# Phase 19 — UI Design Contract

> Visual and interaction contract for Phase 19 (Manual One-Step + Keypad Polish). Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Scope reminder:** Polish + flow consolidation phase. Reuses Phase 18 `TransactionDetailsForm`, existing `AppTextStyles`, existing `AppColors`. The only NEW visual surfaces are `ManualOneStepScreen` (composition) and `KeyboardToolbar` (floating 44dp bar). `SmartKeyboard` is refactored in place (responsive height + spacing).
>
> **Stack:** Flutter / Material 3 — units are **logical pixels (dp / iOS pt)**, not CSS px. Typography tokens reference `lib/core/theme/app_text_styles.dart`; color tokens reference `lib/core/theme/app_colors.dart`. No shadcn (Flutter project).

## Spacing Scale

Declared values (logical pixels / dp; multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4 dp | (legacy SmartKeyboard column gap — replaced by `sm` in Phase 19) |
| sm | 8 dp | (legacy SmartKeyboard row gap — replaced by `md` in Phase 19) |
| md | 12 dp | **SmartKeyboard row gap (between digit rows) — NEW Phase 19 value (was 8)** |
| md+ | 16 dp | Default element spacing; details-section field gap; AppBar→content gap |
| lg | 24 dp | Section padding (between LedgerTypeSelector and DetailInfoCard, etc.) |
| xl | 32 dp | Reserved (not used in Phase 19 layouts) |
| 2xl | 48 dp | **Minimum SmartKeyboard key height floor (iOS HIG 44pt ∪ Material 48dp — pick larger so both platforms pass)** |
| 3xl | 64 dp | Reserved (not used in Phase 19) |

**Phase 19 keypad-specific exceptions (locked by CONTEXT.md D-06/D-07/D-08):**

| Token | Value | Usage |
|-------|-------|-------|
| keypad.colGap | 6 dp | Gap between adjacent keys within a SmartKeyboard row (was 4 dp; +50%) |
| keypad.rowGap | 12 dp | Gap between SmartKeyboard rows (was 8 dp; +50%) |
| keypad.totalHeight | `MediaQuery.size.height × 0.40` | Total SmartKeyboard vertical footprint (responsive ~40% screen) — D-06 |
| keypad.keyHeight | `(totalHeight − rowGaps − safeArea.bottom) ÷ 5` | Per-key height; **MUST clamp to ≥ 48 dp** — D-06 |
| keypad.keyHeight.floor | 48 dp | Hard safety floor (satisfies iOS HIG 44pt AND Material 48dp simultaneously) — SC-2 |
| keyboardToolbar.height | 44 dp | Floating toolbar that rides on top of soft keyboard — D-11 |
| keyboardToolbar.bottom | `MediaQuery.viewInsets.bottom` | Toolbar `Positioned.bottom` — sits exactly above soft keyboard — D-13 |

**Exception rationale (D-06, D-07):** Fixed 48 dp under-serves Pro Max screens; fixed 64 dp over-compresses iPhone SE. Responsive sizing calibrates per device while preserving the 48 dp safety floor that satisfies both platform minimums. Per-key height is verified by widget test under MediaQuery `Size(390, 844)` (iPhone 14 baseline) in `smart_keyboard_test.dart`.

**SC-2 contract (binding):** Every `_DigitKey` AND `_ActionKey` (backspace) AND `_GradientKey` (Save) rendered height ≥ **48 dp** at all supported screen sizes (iPhone SE 4.7" through 6.7" Pro Max). Action row keys use the same responsive height as digit keys (no special-cased 50 dp; remove existing `height: 50` literals in `smart_keyboard.dart` lines 145, 260, 329).

## Typography

All values reference `lib/core/theme/app_text_styles.dart`. Sizes in logical pixels; weights are Flutter `FontWeight` (w400=Regular, w500=Medium, w600=SemiBold, w700=Bold). Line-height encoded in style where present.

**Tokens used in Phase 19 surfaces:**

| Role | Token | Size | Weight | Line Height | Usage in Phase 19 |
|------|-------|------|--------|-------------|-------------------|
| Amount (primary, locked) | `AppTextStyles.amountLarge` | 30 | w700 | 0.9 | `AmountDisplay` digit string (tabular figures) — host renders this above the form |
| Section heading | `AppTextStyles.titleMedium` | 15 | w600 | default | `DetailInfoRow` label (date/category chip label) |
| Body | `AppTextStyles.bodyMedium` | 14 | w500 | default | Merchant `TextField` + note `TextField` input + helper text |
| Label (small caps muted) | `AppTextStyles.labelMedium` | 12 | w600 | default | SmartKeyboard digit numerals + action-row caption (Save / 完了 etc.) |

**Weight discipline:** Phase 19 uses only `w500` (body input) + `w600` (labels/section headings) + `w700` (amount). No w400, no w800. Per template's "2 weights max" rule, Phase 19 effectively uses 2 active weights for chrome (w500 + w600) plus the pre-existing amount weight (w700) inherited from the brand. Amount weight is a brand constraint, not a Phase 19 choice.

**Size discipline:** 4 active sizes in Phase 19 surfaces: 30 (amount), 15 (label), 14 (body), 12 (digit/caption). Matches template's "3-4 sizes" guidance.

**Tabular figures (mandatory):** `AmountDisplay` always renders amount with `fontFeatures: FontFeature.tabularFigures()` — already encoded in `amountLarge`. SmartKeyboard digit keys MUST also render their digit glyph with tabular figures so the visual rhythm matches AmountDisplay. Use `AppTextStyles.labelMedium.copyWith(fontFeatures: [FontFeature.tabularFigures()])` or extract a `digitLabel` constant in `app_text_styles.dart` (planner discretion — D-23 spillover).

**Forbidden in Phase 19:** custom inline `TextStyle(fontSize: …)` literals on any monetary or keypad surface. All amount text → `AppTextStyles.amount*`; all keypad numerals → tabular-figures variant of `labelMedium`. Inline `TextStyle` is only acceptable for genuinely one-off non-numeric, non-branded text inside `KeyboardToolbar` and only if it would otherwise duplicate an existing token verbatim.

## Color

All values reference `lib/core/theme/app_colors.dart` (light) + `AppColorsDark` (dark). Existing app palette — **no new color tokens introduced in Phase 19**. The brand 60/30/10 split below is the inherited app system; Phase 19 surfaces all conform.

|------|-------------|------------|-------------------|

| Secondary (30%) — cards/keys | `AppColors.card` `#FFFFFF` + `AppColors.backgroundMuted` `#F5F4F2` | `AppColorsDark.card` `#252836` + `AppColorsDark.backgroundMuted` `#353845` | `DetailInfoCard` background (`card`); SmartKeyboard digit + action key fill (`backgroundMuted`); KeyboardToolbar background (`card`) |

| Ledger-type accent (semantic, brand) | `AppColors.survival` `#5A9CC8` (生存 blue) / `AppColors.soul` `#47B88A` (悦己 green) | same hexes | `LedgerTypeSelector` selected-state fill + selected-tab indicator on `EntryModeSwitcher` (manual mode active state, current behavior preserved) |

| Text primary | `AppColors.textPrimary` `#1E2432` | `AppColorsDark.textPrimary` `#F0F0F2` | Amount glyph, field input, digit-key glyph |
| Text secondary | `AppColors.textSecondary` `#ABABAB` | `AppColorsDark.textSecondary` `#6B6E7A` | Caption text, KeyboardToolbar "Done" label |
| Destructive | `AppColors.borderInputActive` `#E85A4F` (same coral; doubles as validation-error border) | same | Inline validation toast (`SoftToast`) border + text when amount = 0 or category missing |

**Accent reserved-for list (explicit, exclusive):**
1. SmartKeyboard `_GradientKey` Save button (last cell of 5th row, coral gradient + `actionShadow`).
2. KeyboardToolbar right-hand "记账" / Save compact button (same coral gradient, smaller — D-21).

**Accent forbidden on:** digit keys, backspace key, currency-display middle key, `LedgerTypeSelector` toggles (those use `survival`/`soul`), `EntryModeSwitcher` (uses brand neutrals), `DetailInfoCard` chevrons (uses `textTertiary`), date/category chip backgrounds, merchant/note TextField active border. The coral gradient is the single "Save now" affordance — overusing it dilutes the call-to-action hierarchy.

**Ledger-type semantic colors (carry-over, not accent):** `survival` blue (`#5A9CC8`) and `soul` green (`#47B88A`) are brand semantic colors mapped to ledger type. They are NOT counted as "accent" for the 60/30/10 split — they're semantic state colors on `LedgerTypeSelector` only. The screen's overall chrome accent stays coral (matching the existing `TransactionEntryScreen` pattern users already know).

**Dark mode parity:** Every Phase 19 surface uses `Theme.of(context).brightness == Brightness.dark ? AppColorsDark.* : AppColors.*` lookup. SmartKeyboard already follows this pattern (`smart_keyboard.dart` lines 143-144, 192, 263-264) — preserve verbatim through the refactor.

**Golden-test color invariants (SC-3 enforcement):**
- Digit key fill MUST be exactly `AppColors.backgroundMuted` (light) / `AppColorsDark.backgroundMuted` (dark) — verified by the 6 golden images in `smart_keyboard_golden_test.dart`.
- Gradient Save button MUST be exactly `actionGradientStart → actionGradientEnd` with `actionShadow` drop shadow — golden-verified.
- Key-to-key contrast (digit fill `#F5F4F2` vs screen background `#FCFBF9` in light mode) gives **ΔE ≈ 1.5** which is intentionally subtle (groups keys without "boxy" outlines); the +50% inter-key spacing (D-07) compensates for the low fill contrast to keep keys visually discriminable. Do not "fix" the low key-vs-bg contrast — the spacing carries discriminability per D-07.

### State machine (focus & keyboards)

| State | `_amountFocused` | `_isTextFieldFocused` | SmartKeyboard | Soft keyboard | KeyboardToolbar |
|-------|------------------|------------------------|---------------|---------------|-----------------|
| Initial mount | `true` | `false` | **visible** (slide IN on first frame) | hidden | hidden |
| Tap merchant/note TextField | `false` | `true` | **slides OUT** (220ms easeInOut, offset 0→1) | slides UP (system) | **appears** (gate: `viewInsets.bottom > 0`) |
| Tap toolbar "Done" | becomes `true` (after unfocus) | `false` | **slides IN** | dismisses | disappears |
| Tap toolbar "Record" | unchanged | unchanged | unchanged during transition | unchanged | unchanged → on save, screen pops |
| Tap AmountDisplay (while in TextField) | `true` | `false` (unfocus call) | slides IN | dismisses | disappears |
| Tap SmartKeyboard Save (last cell row 5) | unchanged | unchanged | unchanged | unchanged | n/a → on save, screen pops |

**Transition timing (D-05, D-19):** SmartKeyboard slide uses `AnimatedSlide(offset: Offset(0, showKeypad ? 0 : 1), duration: Duration(milliseconds: 220), curve: Curves.easeInOut)`. Planner may adjust ±50 ms or substitute Material easing curves (`Curves.fastOutSlowIn` is the documented Material entrance curve; either is acceptable). Toolbar appears/disappears via implicit `Visibility(visible: viewInsets.bottom > 0)` — no animation on the toolbar itself; the soft keyboard's own slide animates the toolbar's `bottom` position via `MediaQuery.viewInsets.bottom` changes.

**Touch targets (SC-2 — binding):**
- Every SmartKeyboard cell (digit, action, gradient) renders at **min(responsive, 48 dp floor)** — verified by widget test that queries each cell's `RenderBox.size.height` and asserts `>= 48.0`.
- KeyboardToolbar height = 44 dp; left "Done" tap target is the full left half (≥ 100 dp × 44 dp), right "Record" tap target is the right half (≥ 100 dp × 44 dp) — both satisfy iOS HIG.
- AmountDisplay tap target = its full rendered surface (≥ 60 dp tall in current implementation — preserved).
- `DetailInfoRow` chevrons keep their current ≥ 56 dp row height (Material list item).

**Visual discriminability (SC-3 — binding, golden-test enforced):**
- Adjacent digit keys separated by **6 dp horizontal gap + 12 dp vertical gap** (D-07).
- Digit key fill = `backgroundMuted`; screen background = `background`; deliberate subtle contrast — discriminability carried by spacing, not by border or shadow.
- Backspace key (`⌫`) uses identical fill to digits (peer relationship per D-08).
- Gradient Save button (`_GradientKey`) is the visually distinct row-5 cell — coral gradient + drop shadow makes it the obvious primary CTA.
- Golden test file: `test/widget/features/accounting/presentation/widgets/smart_keyboard_golden_test.dart` × {ja, zh, en} × {light, dark} = **6 golden images**. Isolated SmartKeyboard widget (not whole screen). Existing `flutter_test.matchesGoldenFile` infrastructure; no new test dep.
