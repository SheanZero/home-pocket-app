### 1. WARNING — member-filtered donut pull-to-refresh staleness (post-milestone quick-task, not a v1.8 phase)
`category_donut_card.dart:191` watches `memberFilteredCategoryBreakdownProvider` (keyed on `deviceId`) when a member filter is active, but `categoryDonutRefreshTargets` (`state_analytics.dart:283-315`) does **not** list it. When a member filter is active, pull-to-refresh re-fetches every other donut path but the displayed filtered breakdown serves cached data (the active `ref.watch` keeps the auto-dispose instance alive). Error-retry recovers locally. Introduced by quick-task `260622-d5i`, **after** Phase 47 closed — not a v1.8 phase deliverable; GUARD-01 unaffected (the provider is analytics-only).
- **Fix options:** (a) thread `donutDimensionStateProvider`'s filter into `AnalyticsCardContext` so the refresh-targets function can key `memberFilteredCategoryBreakdownProvider`, or (b) accept auto-dispose retry-only semantics and document. The registry whitelist test verifies union ⊆ whitelist (isolation), not union ⊇ card-watches (completeness) — consider a completeness assertion.
