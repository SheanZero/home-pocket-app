<objective>
Create the two new domain-layer Freezed model files that downstream plans (04 DAO, 05 use case, 06 state, 07-08 widgets) depend on. The load-bearing decision is D-04: `SurvivalLedgerSnapshot` MUST NOT have an `avgSatisfaction` field — this is the type-system enforcement of the engagement-axis re-frame (CONTEXT D-01..D-04 + Pitfall 7 in RESEARCH).

**Canonical naming (revision iteration 1 — resolves checker BLOCKER 2):**
- Domain interchange shape (this plan): `PerCategorySoulBreakdownItem` (D-06 calls it an "item" — each row of a "vertical ranked list, one row per category"). This is the type the repository interface returns and the use cases / widgets consume.
- DAO-tier transient row (Plan 16-04): `PerCategorySoulRowRaw` (a `(categoryId, avgSatisfaction, totalCount)` Drift-row tuple). DAO impl converts `PerCategorySoulRowRaw → PerCategorySoulBreakdownItem` at the repository-impl boundary.
- These two names cannot collide; the domain layer NEVER imports the DAO row type.

Purpose: Lock the data contracts before any DAO/use-case code is written. Once these models exist, any later attempt to compute `AVG(soul_satisfaction)` over survival rows is a compile error (the model literally cannot carry such a field). Equally, any attempt to import `data/daos/analytics_dao.dart` from the domain interface is a CLAUDE.md Pitfall #2 violation (Domain → Data forbidden, enforced by import_guard custom_lint).
Output: Two new Dart files + generated `.freezed.dart` siblings + two Freezed-equality test files. No business logic yet — pure data shape definitions.
</objective>
