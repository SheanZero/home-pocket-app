## Spacing Scale

Declared values (all multiples of 4, sourced from existing AnalyticsScreen card precedents — `family_insight_card.dart`, `category_spend_donut_chart.dart`, `satisfaction_distribution_histogram.dart`):

| Token | Value | Usage in Phase 16 |
|-------|-------|--------------------|
| xs | 4px | Inline icon-to-text gaps inside a row (e.g., category icon → category name) |
| sm | 8px | Row-internal vertical gaps inside a column cell (e.g., between count and spend lines in a Soul-vs-Survival cell) |
| md | 12px | Card title → first content row; between list rows in `_PerCategoryBreakdownCard`; standard `SizedBox(height: 12)` precedent in donut card |
| md+ | 14px | Card padding (`EdgeInsets.all(14)`) and card `BorderRadius.circular(14)` — matches `family_insight_card.dart` exactly |
| lg | 16px | Outer horizontal padding of cards within the Distribution section (existing AnalyticsScreen scaffold — Phase 16 does not change scaffold) |
| xl | 24px | Vertical breathing room between Distribution section cards (existing `_buildSection` precedent — Phase 16 inherits) |

**Exceptions:** none. Phase 16 explicitly does NOT introduce new spacing primitives, divider widths, or non-multiple-of-4 values. The 2×2 grid in `_SoulVsSurvivalCard` (group mode) uses an 8px row-gap and a 1px hairline column divider inside the card; the 1px divider is a stroke, not a spacing unit, and is the project's existing `wmBorderDivider` width default.

## Typography

Phase 16 introduces **zero new text styles**. All four below are existing `AppTextStyles` tokens. The "must declare exactly 3-4 sizes / 2 weights" rule is satisfied: 4 sizes (24/15/14/13), 2 weights (w700 + w500 dominate; w600 appears only on already-defined `titleMedium` reused for sub-row labels).

| Role | Token | Size | Weight | Line Height | Where used in Phase 16 |
|------|-------|------|--------|-------------|-------------------------|
| Card title | `AppTextStyles.titleLarge` | 16px | 700 | default | `_SoulVsSurvivalCard` header ("Ledger · This window" / "本月账本描述" / "今期の家計簿") and `_PerCategoryBreakdownCard` header ("You · Top categories" / "Family · Top categories" in group mode) |
| Cell numeric (primary) | `AppTextStyles.amountMedium` | 18px | 700 (tabular figures) | default | Each Soul-vs-Survival cell's top numeric line (entry count); per-category row trailing `avg sat / N entries` |
| Cell label | `AppTextStyles.bodyMedium` | 14px | 500 | default | Soul/Survival column header labels inside the comparison card ("Soul" / "Survival" / "灵魂" / "生存"); per-category row leading category name |
| Caption / Empty / Other-fold | `AppTextStyles.caption` | 12px | 500 (secondary color) | default | "No data this window" Empty body; "Other: N entries across M categories" fold row; group-mode "Family data not available this window" |

**Font family:** `Outfit` everywhere (inherited from the four styles above). Tabular figures (`FontFeature.tabularFigures()`) are mandatory for all numeric displays — already baked into `amountLarge/Medium/Small`. Phase 16 widgets MUST use the amount styles for any number, per `CLAUDE.md` "Amount Display Style" rule.

**Why no new styles:** D-14 (anti-toxicity copy review) and the Variant ε layout precedent both demand the new cards visually homogenize with `family_insight_card.dart` and `satisfaction_distribution_histogram.dart`. Introducing a 5th size or a w800 would break that homogeneity and add cognitive load with no product justification.

**Forbidden substrings** (must be absent from all rendered output of Phase 16 widgets in all three locales — verified by widget test per D-14):

- **en:** `better`, `worse`, `winner`, `loser`, `vs`, `versus`, `compare`, `comparison`, `higher is good`, `lower is bad`, `score`, `rank`, `ranking`, `wins`, `loses`
- **zh:** `更好`, `更差`, `赢`, `输`, `胜`, `败`, `vs`, `对比`, `比较`, `排名`, `分数`, `胜出`, `落败`

**Rationale (load-bearing):** the entire Phase 16 product framing is **descriptive engagement axis, not satisfaction comparison** (CONTEXT.md D-01..D-04). The Soul column carries an **asymmetric** `avg satisfaction` row by design (D-03) because only Soul has user ratings; Survival NEVER shows a satisfaction-derived number (D-04). Any copy that drifts toward "Soul vs Survival" framing or implies one ledger ranks above the other violates ADR-012 §2/§4/§5 and ADR-014's picker semantics.

## State Matrix (per new widget)

Each widget MUST render all four states. `gsd-ui-checker` validates presence.

### PerCategoryBreakdownCard

| State | Trigger | Visual |
|-------|---------|--------|
| Loading | `AsyncValue.loading` | Existing card skeleton placeholder (reuse `analytics_card_error_state.dart` loading variant or equivalent project precedent) |
| Empty | UseCase returns `Empty` (zero soul entries in window) | Card title + `analyticsPerCategoryEmpty` body, no rows, no expand affordance |
| Sub-min-N only | ≥1 entries but ALL in <3-entry categories | Card title + zero ranked rows + "Other" fold row with aggregate count + zero "Show all" affordance |
| Value (default) | ≥1 category with ≥3 entries | Top-5 ranked rows (sorted `avgSat DESC, count DESC, categoryId ASC` per D-07) + "Other" fold row (if low-N categories exist) + "Show all" affordance (if >5 qualifying categories) |
| Value (expanded) | User tapped "Show all" | All qualifying categories listed in same sort order + "Other" fold row + "Show less" affordance |
| Error | `AsyncValue.error` | Reuse `AnalyticsCardErrorState` (existing widget) with pull-to-refresh hint copy |

### SoulVsSurvivalCard

| State | Trigger | Visual |
|-------|---------|--------|
| Loading | `AsyncValue.loading` | Card skeleton placeholder |
| Empty | EITHER ledger has 0 entries in window (per D-05) | Card title + `analyticsLedgerEmpty` body — no partial population |
| Value — solo mode | Both ledgers have ≥1 entry, `isGroupMode = false` | Card title + two-column layout: left Soul cell (count, spend, avg sat), right Survival cell (count, spend) |
| Value — group mode, family available | Both ledgers ≥1 entry AND `isGroupMode = true` AND `shadowBooks.length ≥ 2` | Card title + 2×2 grid: row 1 "You" (Soul cell + Survival cell), row 2 "Family" (Soul cell + Survival cell) |
| Value — group mode, family unavailable | Both ledgers ≥1 entry, `isGroupMode = true`, BUT `shadowBooks.length < 2` (per D-20) | Card title + 2×2 grid where row 2 Soul and Survival cells render `analyticsLedgerFamilyEmpty` placeholder |
| Error | `AsyncValue.error` | Reuse `AnalyticsCardErrorState` |

**Interaction contract:** Both cards are read-only in Phase 16. Tap behavior on rows, cells, and the "Other" fold row is **noop** (planner discretion noted; recommendation is noop per CONTEXT.md). The "Show all" / "Show less" affordance on `PerCategoryBreakdownCard` is the only interactive element introduced.

## Anti-Pattern Checklist

Phase-16-specific anti-patterns derived from CONTEXT.md and ADR constraints. Each MUST be verified absent during checker and auditor sign-off.

- [ ] No `AVG(soul_satisfaction)` over Survival rows anywhere (DAO, use case, or UI — D-04)
- [ ] No survival-side "satisfaction" number, no Joy/¥ proxy, no rating-derived score on Survival surfaces (D-04)
- [ ] No "comparison" / "vs" / "versus" / "比较" / "対決" framing in card titles or anywhere user-visible (D-12, D-14)
- [ ] No per-family-member breakdown, no per-member labels, no `GROUP BY book_id` projections (ADR-012 §6, D-16)
- [ ] No cross-period delta UI (this-window vs last-window) — ADR-012 §4, D-discretion
- [ ] No celebration animation, toast, snackbar, notification, or haptic when a category leads or values change (ADR-016 §5, 100%-behavior contract extends to Phase 16 surfaces)
- [ ] No HomeHero modification, no HomeHero provider invalidation in `_refresh()` (Phase 15 D-12 binding)
- [ ] No new font families, no new spacing primitives outside the 4-multiple scale above, no new color primitives outside existing `AppColors` / `AppColorsDark`
- [ ] All numeric displays use `AppTextStyles.amountLarge/Medium/Small` (tabular figures mandatory per `CLAUDE.md`)
- [ ] All currency formatting via `NumberFormatter.formatCurrency`; all dates via `DateFormatter`; all UI strings via `S.of(context)`
- [ ] ARB ja/zh/en parity; `flutter gen-l10n` must succeed without warnings
- [ ] Widget tests assert absence of all forbidden substrings in all three locales for both new widgets
