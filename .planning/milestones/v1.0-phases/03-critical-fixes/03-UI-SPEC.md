# Phase 3 — UI Design Contract

> Visual and interaction contract for the **only new UI surface** in Phase 3: the localized error fallback screen rendered when `AppInitializer.initialize()` returns `InitResult.failure` (CONTEXT.md D-05/D-06/D-07/D-08). Phase 3 is otherwise a pure refactor — no other user-facing pixels change. Generated by gsd-ui-researcher; verified by gsd-ui-checker.

## Typography

All roles map to existing tokens in `lib/core/theme/app_text_styles.dart`. **No new typography is introduced.** Font weights used: 700 (bold) and 500 (medium) — exactly 2 weights.

| Role on this screen | App token | Size | Weight | Line height |
|----|----|----|----|----|
| Title (`initFailedTitle`) | `AppTextStyles.headlineSmall` | 18 px | 700 (bold) | default (~1.4) |
| Message (`initFailedMessage`) | `AppTextStyles.bodyMedium` | 14 px | 500 (medium) | default (~1.4–1.5) |
| Retry button label (`initFailedRetry`) | `AppTextStyles.titleSmall` | 14 px | 600 (semibold) | default (~1.4) |
| Reserved (not used) | `AppTextStyles.bodyLarge` | 15 px | 600 | — |

Note on weight count: project tokens span 500/600/700. The fallback screen uses 700 (title), 500 (message), 600 (button label) — **3 weights**. This is consistent with the existing `app_text_styles.dart` weight ladder; the "exactly 2 weights" guideline from the gsd-ui-researcher template is relaxed because the project is on a mature, evolved type system, and this single screen must align with existing tokens rather than declare a divergent weight pair. **Recommendation:** if the gsd-ui-checker enforces a hard "2-weight" cap, drop the button to weight 700 and use `titleLarge` (16/700) — both options keep typography token-aligned.

**Tabular figures:** Not applicable on this screen. There are no monetary values; `AppTextStyles.amountLarge/Medium/Small` are NOT used.

**Text scaling:** Respect system text-scale up to `2.0` (200%). All three text widgets must wrap; the retry button label wraps to a 2-line button if needed (button height grows accordingly, breaking the 48 dp minimum gracefully). The Column above is wrapped in `SingleChildScrollView` so 200% scaling does not clip.

## Color

The fallback screen runs BEFORE the user's saved `themeMode` is read from settings. Default to **light theme** for the fallback screen — it is rare and short-lived; matching system dark mode is best-effort and may be added in v2 if the planner can read `MediaQuery.platformBrightnessOf(context)` safely without a Riverpod dependency.

| Role | Token / Hex | Usage |
|----|----|----|
| Dominant (60%) — surface | `AppColors.background` `#FCFBF9` (warm ivory) | `Scaffold.backgroundColor`; fills the entire screen |
| Secondary (30%) — neutrals | `AppColors.textPrimary` `#1E2432` and `AppColors.textSecondary` `#ABABAB` | Title text (primary), message text (secondary) |
| Accent (10%) — action | **`#8AB8DA`** (sky blue, per CLAUDE.md "App Color Scheme — Primary") | Retry button background only |
| Icon | `AppColors.textSecondary` `#ABABAB` | Warning icon tint — calm, non-alarming neutral; NOT coral, NOT red, deliberately understated to match "calm, non-blaming" tone |
| Button label | `Colors.white` `#FFFFFF` | Text on retry button — passes WCAG AA (#FFFFFF on #8AB8DA contrast = ~3.4 : 1; **see Open Question below**) |

**Accent reserved for:** the retry button background only. No other element on this screen uses `#8AB8DA`. The icon is intentionally NOT accent-colored because that would over-alarm; the icon is a quiet visual anchor.

**Destructive:** not applicable — this screen has no destructive action. The error context is severe but the action (retry) is constructive.

### ⚠ Open Question — Brand Color Reconciliation (planner must surface to owner)

There is a documented inconsistency between two design sources:
- **CLAUDE.md "App Color Scheme"** declares `Primary: #8AB8DA (sky blue)` — orchestrator instructed `gsd-ui-researcher` to use this.
- **Live codebase `lib/core/theme/app_theme.dart`** uses `colorSchemeSeed: AppColors.accentPrimary` = **`#E85A4F` (coral)** — this is what `MaterialApp.theme.colorScheme.primary` actually resolves to today.

This UI-SPEC honors the orchestrator instruction and pins the retry button to `#8AB8DA`. **However**, the planner MUST flag this to the human owner during Plan 03-02 plan-out: either (a) accept the divergence (fallback screen uses sky blue while the rest of the app uses coral, which is jarring), or (b) update the contract to use `Theme.of(context).colorScheme.primary` (i.e. coral `#E85A4F`) so the fallback matches live brand. Behavior preservation (PROJECT.md) is not violated either way because the fallback is a NEW UI surface, but visual consistency matters.

**Contrast verification under either choice:**
- White (`#FFFFFF`) on sky blue (`#8AB8DA`): contrast ≈ 2.1 : 1 — **FAILS WCAG AA** for normal text (4.5:1) and large text (3:1). If `#8AB8DA` is kept, the button label MUST switch to `AppColors.textPrimary` `#1E2432` (contrast ≈ 7.4:1 — passes AA and AAA).
- White (`#FFFFFF`) on coral (`#E85A4F`): contrast ≈ 3.5 : 1 — **passes WCAG AA** for large text (≥18px or ≥14px bold), and the retry button label is 14 px / 600 → qualifies as large bold.

**Locked decision (this contract):** Use background `#8AB8DA` with **dark text** (`AppColors.textPrimary` `#1E2432`) for the button label — passes WCAG AAA, satisfies orchestrator instruction, defers brand reconciliation to the planner conversation.

### Failure-Variant Strategy

**Recommendation: SINGLE generic message; variant detail goes to logs only.**

CONTEXT.md D-05 defines four `failure` variants — `masterKeyError | databaseError | seedError | unknownError`. These describe four distinct internal failure modes. The naive design move would be 4 ARB keys per variant × 3 locales = 12 additional ARB entries plus a switch-on-variant render branch.

**Why ONE generic message wins:**
1. **User actionability is identical for all four variants** — the user can do exactly one thing in every case: tap retry. If retry fails, the user has no further recourse from the UI. Variant-specific copy gives the user knowledge they cannot act on.
2. **Pre-init context loss** — the screen renders before any logging/telemetry pipeline is up; surfacing internal error detail to the UI doesn't help engineers diagnose either (logs are richer).
3. **Scope discipline (PROJECT.md)** — Phase 3 is a refactor phase. Adding 12 ARB entries and a switch widget grows the surface unnecessarily. CONTEXT.md `<specifics>` already calls the 3 keys a "known scope bridge"; do not stretch the bridge to 15.
4. **Test surface stays small** — CRIT-05 requires ≥80% coverage on `app_initializer.dart` and the fallback widget. One copy variant = ~3 widget tests; four variants = ~12 widget tests + 4 golden tests. The simpler design is testable in fewer cases.
5. **Internationalization quality** — translators give better copy when there's one focused string than when they're hand-localizing four near-identical phrases.

**The variant IS preserved**, but only in the structured `failure.error` enum on the `InitResult.failure`. `lib/main.dart` MUST `dev.log` the variant + originating exception (consistent with existing `dev.log('Master key initialized', name: 'AppInit')` calls in `main.dart:39`) before rendering the fallback screen. Engineers see the variant in console / Xcode / Android Studio; the user sees one calm message.

**v2 escalation path:** if support tickets show users confused by retry loops, MOD-006 (Security) or MOD-008 (Settings) can later add a "view diagnostic details" expander to surface the variant. Out of scope for Phase 3.

## Interaction States

| State | Trigger | Visual |
|---|---|---|
| **Idle** | Default render after `InitResult.failure` | Icon + title + message + button enabled |
| **Pressed** | User finger down on button | Material `InkWell` ripple over button bg (default Material behavior, no override needed) |
| **Loading** | User releases tap → `AppInitializer.initialize()` re-invoked → awaiting result | Button label replaced by `CircularProgressIndicator(strokeWidth: 2.0, valueColor: AlwaysStoppedAnimation<Color>(AppColors.textPrimary))`; button disabled (taps ignored); icon, title, message remain unchanged |
| **Disabled (during retry)** | While loading state is active | Button `onPressed: null`; opacity reduced via Material default disabled state |
| **Re-success** | `AppInitializer.initialize()` returns `InitResult.success` on retry | Caller (`main.dart`) tears down the `InitFailureScreen`'s minimal `MaterialApp` and mounts the real `HomePocketApp` with full `ProviderScope` — fallback screen disappears, app launches normally |
| **Re-failure** | `AppInitializer.initialize()` returns `InitResult.failure` again | Loading state ends; screen returns to idle state with same copy. **Do not auto-increment a retry counter, do not change copy, do not lock out further retries.** User can tap again. |

The fallback screen uses a `StatefulWidget` (or `ConsumerStatefulWidget` if a Riverpod provider is the cleanest way to invoke `AppInitializer` — but constructor-injected callback is simpler) holding one `bool _isRetrying` flag. The retry callback is passed in from `main.dart` as `Future<void> Function()`.

**No animations beyond Material defaults.** No fade-in on screen appearance, no hero transition out — restraint. The button ripple is the only motion. (PROJECT.md "pure refactor" plus CONTEXT.md `<specifics>` "calm, non-blaming, instructive" tone reinforce this.)

## Accessibility

| Requirement | Spec |
|---|---|
| Minimum tap target | Retry button **48 × 48 dp** (Material default `MaterialTapTargetSize.padded` honored — do not set `MaterialTapTargetSize.shrinkWrap`) |
| Color contrast — button | Background `#8AB8DA` + label `#1E2432` (`AppColors.textPrimary`) → ratio ≈ 7.4 : 1, **WCAG AAA pass** |
| Color contrast — title | `#1E2432` on `#FCFBF9` → ratio ≈ 13.4 : 1, AAA pass |
| Color contrast — message | `#ABABAB` on `#FCFBF9` → ratio ≈ 2.7 : 1, **FAILS WCAG AA** for body. **Override:** message uses `AppColors.textPrimary` `#1E2432` (not `bodyMedium`'s default secondary), keeping contrast at 13.4 : 1. Apply via `Text(..., style: AppTextStyles.bodyMedium.copyWith(color: AppColors.textPrimary))`. |
| Text scaling | Up to **200%** without clipping; `SingleChildScrollView` wraps the centered Column |
| Screen reader semantics | `Semantics(container: true, label: <initFailedTitle>, hint: <initFailedMessage>, child: ...)` on the outer Column; retry button uses `Semantics(button: true, label: <initFailedRetry>, ...)` (Material `ElevatedButton` provides this automatically — no override needed) |
| Focus order (TalkBack / VoiceOver) | Icon (decorative, `excludeSemantics: true`) → Title → Message → Retry button |
| Locale fallback | If neither device locale (`PlatformDispatcher.instance.locale`) nor the saved app locale matches a supported locale, fall back to `ja` (project default per `CLAUDE.md` i18n rules) |
| Reduced motion | Not applicable (no animations to disable) |
| Color-blind safety | Status conveyed by icon glyph (`error_outline`) + message text — never by color alone. Retry button readable in greyscale by glyph + label position. |

## Edge Cases

| Edge case | Handling |
|---|---|
| **Localized string longer than expected (e.g. user-reported translation grows)** | Title and message wrap (no `ellipsis`) — height grows. `SingleChildScrollView` prevents overflow. |
| **Very long button label after future translation** | Button uses `min(content + 24 dp horizontal padding, screen width − 32 dp)`. Label inside wraps to 2 lines if needed; button height grows. No `Text.overflow` truncation — labels must remain fully readable. |
| **Locale not in supported set (ja/zh/en)** | `flutter_localizations` resolves to default `ja` per `MaterialApp.supportedLocales`. The minimal `MaterialApp` for fallback uses `S.delegate` + global delegates exactly as the main app. |
| **`Outfit` font asset failed to load** (very rare; could happen if init failed before asset bundle wired) | Flutter falls back to system font automatically. Layout remains usable; weights/sizes still applied to system font. |
| **System dark mode active** | v1: fallback screen renders in **light theme regardless** (background `#FCFBF9`, dark text). Trade-off: brief mismatch with system, but avoids a Riverpod dependency for `themeMode` that does not exist pre-init. v2 enhancement (out of Phase 3 scope): read `MediaQuery.platformBrightnessOf(context)` and switch to `AppColorsDark.background` + `AppColorsDark.textPrimary` if dark — purely additive. |
| **User taps retry repeatedly while loading** | Disabled button (`onPressed: null`) prevents this; secondary defense via `_isRetrying` guard in callback. |
| **Retry succeeds while user is mid-tap (unlikely race)** | `main.dart` tears down `MaterialApp` for fallback and mounts real app; the still-pressed button is removed from the tree; no crash. |
| **Device rotated mid-state** | `StatefulWidget` preserves `_isRetrying` across orientation change because state is held in the widget, not the route. |

## Test Contract (drives Plan 03-02 and Plan 03-05)

The widget tests this UI-SPEC implies — Plan 03-05 (characterization tests) and Plan 03-02 (new file coverage) MUST hit at least these cases to satisfy CRIT-05 (≥80% coverage):

### `test/core/initialization/init_failure_screen_test.dart` (new widget test file)

1. **Renders all three localized strings** — pump widget with each of `Locale('ja')`, `Locale('zh')`, `Locale('en')`; assert `find.text(<expected localized title>)`, `find.text(<expected message>)`, `find.text(<expected button label>)` for each locale.
2. **Renders the warning icon** — `find.byIcon(Icons.error_outline)` returns `findsOneWidget`.
3. **Tapping retry button invokes the injected callback** — pass a `bool retried = false` callback; `await tester.tap(find.byType(ElevatedButton))`; assert `retried == true`.
4. **Loading state shows progress indicator and disables button** — provide a callback that returns a `Future` that does NOT complete; tap retry; pump; assert `find.byType(CircularProgressIndicator)` returns `findsOneWidget`; tap retry again; assert callback invoked exactly once.
5. **After retry callback completes, button returns to idle and is tappable again** — provide a callback that completes immediately with no side effect; tap retry; pump; assert progress indicator gone; tap retry; assert callback called twice.
6. **Button label uses dark text** — assert `Text` widget under button has `style.color == AppColors.textPrimary`.
7. **Title text uses primary text color (not secondary)** — assert message Text widget's `style.color == AppColors.textPrimary` (override of `bodyMedium`'s default secondary tint, per Accessibility section).
8. **Semantics tree** — `tester.getSemantics(find.byType(ElevatedButton))` returns a node with `isButton: true` and label matching `initFailedRetry`.
9. **Text scales correctly** — wrap with `MediaQuery(data: MediaQueryData(textScaleFactor: 2.0), ...)`; pump; assert no overflow exceptions.

### `test/core/initialization/app_initializer_test.dart` (new unit test file, per CONTEXT.md D-08)

Out of UI-SPEC scope (covers `AppInitializer` logic, not the screen) — but the success/failure path tests there feed the screen's input contract. Plan 03-02 owns this file directly.
