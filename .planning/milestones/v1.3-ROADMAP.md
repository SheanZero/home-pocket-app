## Overview

v1.3 transformed ledger entry from "multi-step, mis-tap-prone, voice-unreliable" into a "single-screen, accurate, voice-trustworthy" core experience, and reused the same details form as the edit entry-point for existing transactions. Six phases delivered the shared details form, single-screen manual entry with polished keypad, locale-aware voice number parsing (zh + ja), Level-2 category enforcement, voice one-step integration with record-button UX, and a cleanup phase absorbing carried tech-debt.

### Phase 18: Shared Details Form Foundation

**Goal**: Consolidate the entry-details surface into one Freezed-backed widget that serves manual single-screen, voice single-screen, OCR two-step, and edit-existing flows — with `entry_source` preservation on edit and the OCR architectural slot reserved.
**Depends on**: Nothing (first v1.3 phase; foundation for 19/22)

**Success Criteria**:
1. A single `TransactionDetailsForm` widget renders all editable fields and is consumed by manual, voice, OCR two-step, and edit-existing paths — same widget configured via mode parameter (`new` vs `edit`) and optional `Transaction` seed.
2. User can tap any existing transaction in the home recent-tx list and the shared details form opens pre-populated; cancel returns to the list with no DB write.
3. User can modify any field, tap save, and the underlying Drift row is updated atomically; `entry_source` value present before the edit is preserved verbatim (does not flip to `'manual'`).
4. OCR flow code path exposes a two-step container whose step 2 mounts the same shared widget; no OCR writer is implemented — only the architectural slot is reserved.
5. No new Drift schema migration introduced; current v17 schema suffices for edit/save path.

### Phase 21: Voice Category Resolver Level-2 Enforcement

**Goal**: Guarantee voice-driven Transactions always carry a level-2 category by enforcing the always-level-2 contract — falling back to a level-1's first level-2 sub-category when no exact level-2 match exists — and make the resolution data sources extensible without code changes.
**Depends on**: Phase 20 (shares test infrastructure)
**Requirements**: VOICE-04, VOICE-05, VOICE-06
**Success Criteria**:
1. Voice category resolver returns a level-2 category whenever the spoken phrase matches any level-2 entry in the merchant database or synonym dictionary.
2. When voice resolves only to a level-1 category, the resolver returns that level-1's first level-2 sub-category; resulting `Transaction.categoryId` always references a level-2 row.
3. Resolver consults both merchant database AND synonym dictionary before fallback; lookup order documented and verified.
4. Both data sources extensible by adding entries without modifying resolver code; runtime extensibility test for 珍珠奶茶 (zh) / タピオカ (ja).
5. `flutter analyze` 0 issues; per-file coverage ≥70%; resolver in `lib/application/voice/` per "Thin Feature" rule.
